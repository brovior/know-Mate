# KnowMate RAG 전체 구조 검토

## 현재 구조

```
[인덱싱] 파싱 → 청킹 → 임베딩 → 벡터DB 저장
[검색]   질문 → 임베딩 → 유사도검색 → LLM → 응답+출처
```

---

## 추가 검토 필요한 것들

### ① 수집 트리거가 없어
```
언제 파싱을 시작하느냐?
→ 최초 실행 시 전체 스캔
→ 이후 증분 스캔 (파일 변경 감지)
→ 삭제된 파일 orphan 정리
```
> CLAUDE.md 3장 수집기(scanner/scheduler)가 이 역할

---

### ② 메타데이터 저장이 명시 안 됨
```
임베딩만 저장하는 게 아니라
→ 파일경로, 수정일, 파일타입, 권한(scope) 도 같이 저장
→ 검색 후 출처 카드 렌더링에 필수
```

**state.json 스키마**
```json
{
  "C:/문서/A.docx": {
    "mtime": 1000.123,
    "size": 2048,
    "indexed_at": "2026-06-20T11:00:00",
    "chunk_ids": ["uuid1", "uuid2"]
  }
}
```

---

### ③ 검색 후 권한 필터가 없어
```
유사도검색 → (권한필터) → 청크추출
→ 내 PC 문서 / 공유폴더 구분
→ 사이드바 검색범위 체크박스와 연동
```

---

### ④ LLM 앞단 프롬프트 조립 전략
```
검색된 청크가 너무 많으면 컨텍스트 초과
→ Top K 몇 개로 자를지
→ 청크 순서 어떻게 배열할지
→ 시스템 프롬프트 어떻게 짤지
```

---

## 완성된 큰 구조

```
[인덱싱]
수집트리거 → 파싱 → 청킹 → 임베딩 → 메타데이터+벡터 저장
                                      ↑
                               orphan 정리 (증분)

[검색]
질문 → 임베딩 → 유사도검색 → 권한필터 → Top K 추출
     → 프롬프트 조립 → LLM → 응답+출처 반환
```

---

## 설계 결정 우선순위

| 순서 | 항목 | 이유 |
|---|---|---|
| 1 | 메타데이터 스키마 확정 | 나중에 바꾸면 전체 재인덱싱 필요 |
| 2 | 청킹 전략 확정 | 마찬가지로 재인덱싱 필요 |
| 3 | 권한 필터 구조 | 벡터DB 저장 스키마에 영향 |
| 4 | Top K / 프롬프트 전략 | 검색 품질에 직접 영향 |

## 벡터DB 결정

| 항목 | ChromaDB | LanceDB (채택) |
|---|---|---|
| 로컬 퍼시스턴트 | ✅ | ✅ |
| soft delete | 직접 구현 | 자연스럽게 지원 |
| 컬럼 암호화 | 직접 구현 | 컬럼 단위 적용 쉬움 |
| compact/cleanup | 없음 | `compact_files()` 내장 |

> 검증은 ChromaDB로 했으나 CLAUDE.md orphan 안전장치 요구사항(soft delete, compact) 구현 편의상 LanceDB로 전환
