## 총평
두 변경 모두 문제 원인에 직접 대응하며, 종료 경로 수렴과 purge projection·주기적 reconciliation 선택은 전반적으로 타당하다. 특히 실패·차단 상태를 성공 스킵보다 먼저 판정하고 projection 미지원 시 조용히 전체 로드로 폴백하지 않는 결정에 동의한다. 다만 A-0002 내부에 메타 상태 처리 규칙과 정면으로 충돌하는 실패 모드·검증 계획이 남아 있어 구현 및 테스트의 기준이 모호하다. 또한 R-0001의 3초 수용 기준은 실행 중인 정상 워커가 있는 경우의 8초 graceful 대기와 범위가 겹쳐, 동일한 실행이 요구를 만족하면서 동시에 실패할 수 있다.

**판정**: REQUEST_CHANGES

## 동의하는 결정
- 트레이 앱에서 `quitOnLastWindowClosed(False)`와 모든 종료 경로가 수렴하는 명시적 `quit()`을 함께 사용하는 결정은 창 가시성에 따른 Qt 암묵 동작을 제거하므로 적절하다.
- 워커 상태를 확인한 뒤 `quit`과 `hard_exit` 중 하나만 선택하고, 앞 단계 예외가 최종 종료 판정을 건너뛰지 못하게 한 구조는 기존 행오버 에스컬레이션과 역할이 명확히 분리되어 있다.
- purge가 필요한 컬럼만 projection하고 Arrow를 직접 순회하는 결정은 관측된 메모리 문제의 직접 원인을 제거한다. 단순히 주기만 늘리거나 `gc.collect()`에 의존하지 않은 것도 타당하다.
- 실패·차단 억제 → 정상 성공 스킵 → 실행 순으로 판정하고, 실패 시 `reconciled_sig`를 해제하는 상태 전이는 백오프 만료 후 재시도가 성공 메타에 가로막히는 문제를 방지한다.
- sidecar에 경로 원문 대신 서명과 시각만 저장하므로 기존 state 스키마를 건드리지 않으며 추가적인 민감 경로 노출도 만들지 않는다. projection 미지원 시 전체 로드 폴백을 금지한 것도 성능 불변식을 지키는 합리적인 배포 게이트다.

## 지적 사항

### [M-1] A-0002의 실패 모드와 검증 계획이 결정된 메타 상태 전이와 충돌함 (Major)
- **관점**: 1. 요구사항 누락, 7. 동시성, 8. 장애 대응
- **지적**: 문서 내에서 `next_retry_ts`, purge 실패 및 대량삭제 차단 이후의 기대 동작이 서로 다르게 정의되어 있다. 이 상태로 구현하면 본문과 테스트 중 어느 쪽을 정본으로 삼느냐에 따라 백오프나 차단 불변식이 깨질 수 있다.
- **근거**:
  - 데이터 흐름과 ADR-0002는 유효한 미래 `next_retry_ts`를 정상 백오프 상태로 보고 DB 조회 없이 반환한다고 정의한다.
  - 반면 A-0002 실패 모드의 `시스템 시각 역행·미래 시각 필드` 행은 `미래 next_retry_ts`를 일괄적으로 무시하고 purge를 실행하는 것으로 읽힌다. 이는 `now < 값 ≤ now+설정백오프`는 정상이라는 앞선 정의와 충돌한다.
  - 검증 계획 ③은 “purge 예외·대량삭제 차단 시 meta 미갱신 → 다음 사이클 재실행”이라고 되어 있다. 그러나 결정 본문은 실패 시 `failed_sig`와 `next_retry_ts`를 즉시 갱신해 백오프하고, 차단 시 `blocked_sig`를 갱신해 동일 `op_sig`에서는 재실행하지 않는다고 명시한다.
  - R-0002 NFR-3과 AC-4도 매 사이클 재실행이 아니라 백오프·차단 억제를 요구한다.
- **대안**:
  1. 실패 모드를 다음처럼 분리한다: `last_purge_ts`의 모든 미래값은 무효, `next_retry_ts`는 정의된 범위의 미래값이면 정상 억제, 범위를 벗어난 값만 손상으로 취급한다.
  2. 검증 계획 ③을 “실패 메타 갱신 후 백오프 동안 조회 없음, 만료 후 1회 재시도”와 “차단 메타 갱신 후 동일 `op_sig`에서 조회 없음, `op_sig` 변경 후 재실행”의 두 테스트로 교체한다.
  3. “meta 미갱신”이 `reconciled_sig` 미갱신만 의미했다면, `실패·차단 메타는 갱신하되 성공 메타는 갱신하지 않음`으로 정확히 표현한다.

### [M-2] 3초 종료 수용 기준과 8초 graceful 대기의 적용 범위가 불명확함 (Major)
- **관점**: 1. 요구사항 누락, 2. 숨은 가정, 8. 장애 대응
- **지적**: AC-2와 실기 검증의 3초 기준이 워커 미실행 상태에만 적용되는지 명시되지 않았다. 워커가 정상 실행 중이면 `stop_worker`가 최대 8초를 기다릴 수 있으므로, 행오버가 아니더라도 3초 기준을 위반할 수 있다.
- **근거**:
  - AC-1은 명시적으로 “유휴 — 워커 미실행”이라고 한정하지만 AC-2는 창 표시 상태와 `close_action=quit` 경로에 대해 단순히 “동일(3초 이내)”이라고만 한다.
  - A-0001의 실기 계획도 첫 세 경로에 대해 3초를 요구하지만 워커 상태를 고정하지 않는다.
  - 정상 종료 계약은 정상적으로 오래 걸리는 추출에도 8초 graceful 대기가 만료될 수 있다고 인정한다. 따라서 창 가시성은 종료 경로이고 워커 실행 여부는 종료 시간인데, 현재 기준은 두 축을 분리하지 않는다.
- **대안**:
  1. AC-1·AC-2와 첫 세 실기 경로를 모두 `워커 미실행 상태`로 한정하고, 워커 실행 중인 정상 취소 경로에는 별도 상한을 둔다.
  2. 또는 모든 일반 종료를 3초로 유지해야 한다면 graceful 대기 시간을 3초 예산 안으로 줄이고 데이터 무결성·정상 장기 추출에 미치는 트레이드오프를 다시 승인받는다.
  3. 테스트 매트릭스를 `창 숨김/표시/close_action` × `워커 미실행/정상 실행/행오버`로 명시해 각 셀의 시간 상한을 고정한다.

### [m-1] 강제 종료 시 손실 범위를 state 갱신으로만 한정한 근거가 부족함 (Minor)
- **관점**: 2. 숨은 가정, 7. 동시성, 8. 장애 대응
- **지적**: A-0001은 `terminate`/`os._exit` 경로에서 “잃는 것은 그 사이클의 state 갱신뿐”이라고 단정하지만, 전달된 문서만으로는 강제 종료 시점에 LanceDB 쓰기·삭제·`optimize()`가 진행 중이지 않다는 보장이 확인되지 않는다.
- **근거**: QThread 강제 종료나 `os._exit`는 `finally`, flush 및 정상 라이브러리 정리를 건너뛸 수 있다. R-0001 NFR-1은 정상 종료에만 적용되므로 설계 선택 자체가 요구 위반은 아니지만, 비정상 경로의 영향 범위를 state로만 한정하려면 DB 연산의 원자성 또는 재시작 복구 근거가 필요하다.
- **대안**:
  1. 해당 문장을 “최소한 state 갱신이 유실될 수 있으며, DB 연산 중 강제 종료의 영향은 LanceDB 원자성 확인 필요”로 완화한다.
  2. 인덱싱 단계별 강제 종료 후 재시작·조회·재인덱싱 테스트를 추가한다.
  3. LanceDB가 해당 쓰기 경로에서 원자 커밋을 보장한다면 사용 API와 버전을 근거로 문서화하고, 시작 시 DB open 실패에 대한 진단·복구 경로도 명시한다.

### [m-2] sidecar 컴포넌트 책임 표가 실제 상태 스키마를 모두 반영하지 않음 (Minor)
- **관점**: 3. 아키텍처 결함, 8. 장애 대응
- **지적**: A-0002 컴포넌트 표에는 sidecar가 `reconciled_sig`와 `last_purge_ts`만 보관한다고 되어 있지만 실제 데이터 흐름에서는 `failed_sig`, `next_retry_ts`, `blocked_sig`도 영속화한다.
- **근거**: 실패·차단 억제는 NFR-3과 AC-4를 만족시키는 핵심 책임이므로 단순 부가 필드가 아니다. 컴포넌트 표만 보고 구현하거나 스키마를 검토하면 재시작 후 억제 상태가 누락될 수 있다.
- **대안**: 컴포넌트 표에 전체 필드와 각 필드의 의미를 기재하고, sidecar 스키마 버전 및 필드별 유효성 규칙을 함께 링크한다.

## 확인 필요
- 배포 고정 LanceDB 버전에서 `table.to_lance().to_table(columns=["file_path"])` 계열 API가 공개·안정 API인지, 그리고 실제 스캔 단계에서 vector/text 컬럼을 읽지 않는지 확인이 필요하다. 이는 문서에 적힌 대로 채택 전제조건으로 유지해야 한다.
- `max_delete_ratio`에 NaN, Infinity, 음수 또는 1 초과 값이 들어오지 않도록 설정 로더에서 검증하는지 확인이 필요하다. 검증이 없다면 canonical JSON 서명과 차단 판정 모두 비정상 설정의 영향을 받을 수 있다.
- QThread 강제 종료 또는 `os._exit` 시점의 LanceDB 쓰기 원자성과 재시작 복구 동작을 확인해야 한다. 현재 전달된 문서에는 이를 입증할 소스 코드나 테스트 결과가 없다.
