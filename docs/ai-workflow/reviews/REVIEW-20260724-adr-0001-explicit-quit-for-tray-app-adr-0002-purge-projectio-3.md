## 총평
두 변경 모두 관측된 문제의 원인과 해결 방향을 합리적으로 연결했으며, 특히 명시적 Qt 종료와 LanceDB 컬럼 projection은 적절한 선택이다. 다만 purge 설계의 재시도 판정 순서가 현재 의사코드대로라면 백오프가 실제로 작동하지 않아 승인된 NFR을 직접 위반한다. 또한 대량삭제 차단 시 무기한 미복구하는 정책과 “최대 24시간 내 복구” 요구가 서로 양립하지 않으므로 요구사항 의미를 먼저 확정해야 한다. 종료 설계도 예외 발생 시 데이터 무결성과 프로세스 종료 중 무엇을 우선하는지 문서 내부가 모순된다.

**판정**: REQUEST_CHANGES

## 동의하는 결정
- 트레이 앱에서 `quitOnLastWindowClosed(False)`와 명시적 `QApplication.quit()`을 함께 사용하는 결정에 동의한다. 창 가시성에 따른 Qt의 암묵 종료 조건을 제거하고 종료 의도를 한 경로에 모으는 방식이 현재 문제에 직접 대응한다.
- `quit()`을 워커 종료 뒤에 배치하고 기존 `stop_worker` 에스컬레이션을 유지하는 방향은 정상 종료와 행오버 복구를 분리한다는 점에서 타당하다.
- purge에서 `file_path`만 projection하고 Arrow 컬럼을 직접 순회하는 결정에 동의한다. 벡터와 암호문을 전건 materialize하는 현재 비용을 원인 수준에서 제거한다.
- 성공한 reconciliation과 실패한 시도를 구분하고, 메타 부재·손상·시각 역행 시 보수적으로 purge를 실행하는 정책은 재실행 안전성 측면에서 적절하다.
- 기존 `index_state.json` 스키마를 변경하지 않고 별도 sidecar를 원자 교체하는 선택은 베타 단계의 마이그레이션 위험을 줄이는 실용적인 결정이다.
- projection 불가 시 조용히 전체 로드로 폴백하지 않겠다는 원칙과 실제 컬럼 pushdown을 검증하려는 AC-1도 적절하다.

## 지적 사항

### [B-1] 일시적 실패 백오프가 현재 판정 순서에서는 작동하지 않음 (Blocker)
- **관점**: 7. 동시성, 8. 장애 대응
- **지적**: A-0002의 의사코드는 정상 스킵 조건에 `and 백오프 미해당`을 넣고, 조건이 거짓이면 purge를 실행한다. 실패 시 `reconciled_sig`는 갱신되지 않으므로 다음 사이클에는 서명 일치 조건부터 거짓이 되어 백오프 여부와 관계없이 다시 전건 조회하게 된다.
- **근거**: 문서의 흐름은 `if op_sig == reconciled_sig and 처리 0건 ... and 백오프 미해당: skip; else: purge`이다. 일시적 예외 뒤에는 `failed_sig=op_sig`만 기록되고 `reconciled_sig`는 이전 값이므로 매분 `else`에 진입한다. 이는 R-0002 NFR-3과 AC-4의 “다음 사이클마다 전건 조회가 반복되지 않음”을 직접 위반한다.
- **대안**: 실패 억제 판정을 정상 성공 스킵보다 먼저 독립적으로 수행한다. 예:
  1. `failed_sig == op_sig && failure_kind == transient && now < next_retry_ts`이면 DB 조회 없이 반환
  2. `blocked_sig == op_sig`이면 DB 조회 없이 차단 상태 유지
  3. 그다음 `reconciled_sig == op_sig && processed_count == 0 && force_period 미경과`를 평가
  4. 나머지만 purge 실행  
  `next_retry_ts`를 직접 저장하거나 현재 설정으로 계산하는 방식도 함께 명시하고, 이 순서를 단위 테스트로 고정해야 한다.

### [B-2] 대량삭제 차단 정책과 최대 24시간 복구 요구가 상충함 (Blocker)
- **관점**: 1. 요구사항 누락, 8. 장애 대응
- **지적**: 동일 `op_sig`에서 대량삭제가 차단되면 자동 재시도하지 않는다는 결정은 안전하지만, R-0002 FR-3의 “state-DB 불일치 복구는 최대 24시간 지연” 및 A-0002의 강제 reconciliation 상한과 양립하지 않는다.
- **근거**: 차단율이나 구성이 바뀌지 않으면 동일 서명은 영구적으로 재시도되지 않는다. 따라서 24시간이 지나도 실제 복구는 물론 재검사조차 수행되지 않을 수 있다. 일시적 장애가 24시간 이상 지속되는 경우에도 실제 복구 상한을 보장할 수 없어, 현재 요구 문언 자체가 장애·안전 차단 조건을 고려하지 않았다.
- **대안**: 다음 중 하나로 계약을 명확히 정리해야 한다.
  - FR-3을 “정상 상태에서는 24시간 이내 reconciliation을 **시도**하며, 대량삭제 차단·지속 장애 시 실제 삭제는 보장하지 않는다”로 수정한다.
  - 또는 차단 상태에서도 24시간마다 경로 projection과 안전성 재평가만 수행하되, 자동 삭제는 계속 막고 운영자 확인을 요구한다.
  - 실제 삭제까지 24시간 상한을 유지해야 한다면, 대량삭제 차단을 자동 해제할 명시적이고 안전한 승인 절차가 필요하다. 단순 시간 경과만으로 차단을 푸는 방식은 권장하지 않는다.

### [M-1] 종료 단계 예외 시 정상 종료 계약과 `quit()` 보장 정책이 모순됨 (Major)
- **관점**: 1. 요구사항 누락, 8. 장애 대응
- **지적**: A-0001은 `quit()`이 scheduler 정지와 state 저장을 “보장한 뒤에만” 호출된다고 설명하지만, 실패 모드에서는 각 단계 예외와 무관하게 `quit()`을 finally에서 실행한다고 한다. 두 정책은 동시에 성립하지 않는다.
- **근거**: `scheduler.stop()`이 실패하거나 `stop_worker()`가 예외를 던졌는데도 `quit()`을 호출하면 R-0001 NFR-1의 “state 저장·스케줄러 정지 후 종료”를 보장할 수 없다. 특히 `stop_worker` 예외 후 Qt 루프만 종료하면 실행 중 QThread의 종료 여부와 state 저장 여부가 불명확하다. 또한 8초 타임아웃은 COM 행오버뿐 아니라 정상적으로 오래 걸리는 추출·임베딩에도 발생할 수 있으므로, 강제 경로가 항상 “이미 멈춘 워커”에만 적용된다는 설명도 확인이 필요하다.
- **대안**: 단계별 실패 정책을 명시한다. 예를 들어 scheduler 정지 실패는 이벤트 루프 종료로 신규 Qt 타이머가 더 실행되지 않는다는 근거를 남기고, worker가 여전히 실행 중이거나 `stop_worker` 자체가 실패하면 명시적인 최종 `hard_exit` 경로로 전환한다. 정상 경로에서만 “state 저장 후 종료”를 보장하고, 강제 경로는 데이터 무결성 예외로 별도 기술해야 한다. 테스트에는 scheduler 예외, stop_worker 예외, 정상이나 8초보다 긴 작업을 각각 포함하는 것이 좋다.

### [M-2] projection API와 미지원 시 동작이 채택 가능한 수준으로 확정되지 않음 (Major)
- **관점**: 2. 숨은 가정, 5. 성능, 8. 장애 대응
- **지적**: 핵심 구현 API가 “`table.to_lance().to_table(columns=...)` 계열을 검토”하는 수준에 머물러 있고, 미지원 시의 동작도 요구사항과 완전히 정합되지 않는다.
- **근거**: ADR은 API 확인을 채택 조건으로 두지만 architecture는 아직 `lancedb 0.6+`라는 넓은 범위만 제시한다. 또한 미지원 시 “purge를 구성 변경 사이클로 한정”한다고 했는데, 이 경우 파일 변경 시 즉시 실행 및 24시간 강제 reconciliation 요구가 누락된다. 그 제한된 purge가 전체 로드를 사용하는지, purge 자체를 비활성화하는지도 분명하지 않다.
- **대안**: Accepted 전 배포에 고정된 정확한 LanceDB 버전과 호출 API를 확정하고, 실제 스캔 컬럼 또는 메모리 실측 테스트를 통과시킨다. 지원되지 않는 버전에서는 ① 해당 변경의 배포를 중단하거나, ② 요구를 만족하는 다른 공식 projection API를 사용해야 한다. 불가피하게 전체 로드를 사용하는 호환 모드를 둔다면 그 적용 조건·빈도·경고·24시간 정책을 요구사항에 명시적으로 반영해야 한다.

### [m-1] op_sig 경로 정규화 계약이 Windows 경로 별칭을 충분히 정의하지 않음 (Minor)
- **관점**: 2. 숨은 가정, 9. 운영 복잡도
- **지적**: `normcase/normpath·구분자 통일`만으로는 상대 경로, 드라이브 문자와 UNC 별칭, 후행 구분자, 환경변수 확장 등 실제 동일 폴더의 여러 표현을 어떻게 처리할지 불명확하다.
- **근거**: 잘못된 정규화는 삭제 정확성보다는 불필요한 op_sig 변경과 purge 재실행을 유발할 가능성이 크다. 반대로 서명 계산과 실제 소속 판정이 서로 다른 정규화 함수를 사용하면 더 심각한 불일치가 생길 수 있다.
- **대안**: 설정 로드 후 절대 경로화, 구분자·후행 구분자 처리, Windows 대소문자 정책을 하나의 공용 함수로 정의하고 서명과 purge 소속 판정에서 같은 결과를 사용한다. UNC/매핑 드라이브를 동일 실체로 해석할지 여부는 명시적으로 결정하고 테스트한다.

### [m-2] sidecar의 의미적 손상과 부분 완료 재실행 계약이 부족함 (Minor)
- **관점**: 7. 동시성, 8. 장애 대응
- **지적**: 파일 자체의 JSON 손상은 다루지만, 필드 누락·잘못된 타입·NaN/무한대·미래 `last_attempt_ts` 등 의미적으로 잘못된 메타 처리와 purge 성공 후 메타 저장 실패 시의 재실행 계약은 구체적이지 않다.
- **근거**: DB 삭제 완료 뒤 sidecar 교체가 실패하면 다음 사이클에 같은 purge가 다시 실행된다. 이 동작은 삭제와 optimize가 멱등적이라는 전제가 있어야 안전하다. 미래 시각 방어도 `last_purge_ts`에는 있지만 백오프용 시각에는 명시되지 않았다.
- **대안**: sidecar 스키마 버전과 필드별 타입·범위 검증을 정의하고, 이상 값은 메타 부재와 동일하게 취급한다. 삭제·optimize 재실행의 멱등성을 문서화하고, DB 변경 성공 후 메타 저장 실패 테스트 및 미래 `next_retry_ts` 처리 테스트를 추가한다.

## 확인 필요
- 실제 소스 코드가 제공되지 않아 현재 트레이 [종료]와 `close_action=quit` 경로가 모두 정확히 `closeEvent → _shutdown()`으로 수렴한다는 주장과 `_shutdown()`의 기존 예외 처리 구조는 확인이 필요하다.
- `stop_worker`의 8초 타임아웃이 정상적으로 오래 걸리는 파일에도 적용되는지, `terminate()` 또는 예외 이후 worker가 실제로 정리됐음을 어떻게 판정하는지 확인이 필요하다.
- 배포에 고정된 정확한 LanceDB 버전에서 선택한 projection API가 벡터·암호문 컬럼을 실제로 읽지 않는지 실측 확인이 필요하다.
- 기존 `_purge_removed_folders`가 부분 삭제 후 재실행되어도 안전한지, 삭제와 `optimize()`가 멱등적인지 확인이 필요하다.
