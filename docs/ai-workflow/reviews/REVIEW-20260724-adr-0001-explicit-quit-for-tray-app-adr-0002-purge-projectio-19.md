## 총평
두 ADR의 핵심 방향인 명시적 Qt 종료, purge 컬럼 projection, 보수적 상태 전이와 sidecar 분리는 타당하며 기존 문제의 직접 원인에 잘 대응한다. 특히 강제 종료 이후의 무결성을 과장하지 않고 dirty marker와 수동 복구 경로를 둔 점도 현실적이다. 다만 `app.exec()` 반환만으로 정상 종료를 확정하는 조건과 projection API의 영구적 호출 호환성 실패를 일시 장애로만 처리하는 정책에는 운영상 재설계가 필요한 공백이 있다. 소스 코드와 requirements.md가 제공되지 않아 문서에 선언된 구현·요구사항 대응의 실제 충족 여부는 별도 확인이 필요하다.

**판정**: REQUEST_CHANGES

## 동의하는 결정
- `quitOnLastWindowClosed(False)`와 `_shutdown()`의 명시적 최종 판정을 조합한 것은 트레이 앱의 창 가시성에 종료 여부가 종속되는 문제를 직접 제거하는 적절한 결정이다.
- `QThread.terminate()` 사용 이력을 별도로 전달해 이후 `isRunning()==False`여도 정상 종료로 분류하지 않는 정책에 동의한다. 스레드가 임의 지점에서 중단되면 락과 저장소 상태를 신뢰할 수 없다는 근거가 충분하다.
- hard-exit 경로에서 로깅이나 marker 파일 I/O를 수행하지 않는 결정은 “최후 안전망은 블록되면 안 된다”는 우선순위와 일관된다.
- purge에서 `file_path`만 projection하고 Arrow를 직접 순회하는 것은 메모리 문제의 근본 원인인 벡터·암호문 로드를 제거한다. 스킵만 도입하는 대안보다 확장성이 좋다.
- 성공·실패·차단 메타를 저장 성공 여부와 무관하게 메모리에 먼저 반영하고, 저장 실패 후 재시작 시 보수적으로 재실행하는 설계는 단일 워커 및 멱등 purge 전제에서 합리적이다.
- unsupported 억제 키를 폴더 구성인 `op_sig`가 아니라 앱·LanceDB·projection 전략을 포함한 `capability_sig`로 분리한 것도 장애 원인과 복구 조건을 올바르게 대응시킨 결정이다.

## 지적 사항

### [M-1] `app.exec()` 반환은 정상 `_shutdown()` 완료의 충분조건이 아니다 (Major)
- **관점**: 1. 요구사항 누락, 7. 동시성, 8. 장애 대응
- **지적**: dirty-shutdown marker를 지우는 조건이 단순히 `app.exec()`가 반환했다는 사실뿐이다. 그러나 이벤트 루프 반환은 `_shutdown()`의 정상 분기에서 `quit()`이 호출됐음을 증명하지 않으며, 다른 코드의 `QApplication.quit()` 호출, OS 로그오프·세션 종료 등 별도 종료 원인으로도 발생할 수 있다.
- **근거**: A-0001은 “`app.exec()` 반환 = 정상 quit 확정”이라고 명시하고 반환 직후 marker를 지운다. 동시에 모든 종료 경로가 `_shutdown()`을 경유해야 한다고 선언하지만, 문서가 검증한 경로는 트레이 종료와 `close_action=quit`뿐이며 OS 종료나 외부 이벤트 루프 종료는 다루지 않는다. 이런 경로에서 워커 정리가 완료되지 않은 채 이벤트 루프가 반환하면 marker가 지워져 다음 실행에서 false negative가 발생할 수 있다.
- **대안**: `_shutdown()`의 정상 분기가 워커 비실행을 확인하고 `quit_fn()`을 호출하기 직전에 프로세스 내 `clean_shutdown_requested=True` 같은 플래그를 설정하고, `main()`은 `app.exec()` 반환과 해당 플래그가 모두 참일 때만 marker를 지워야 한다. OS 세션 종료도 정상 정리가 필요한 요구라면 Qt의 세션 종료 신호 또는 중앙 lifecycle 진입점을 통해 `_shutdown()`으로 수렴시키고, 시간 내 정리 실패 시 marker를 남기는 정책을 별도로 정의한다.

### [M-2] 메서드가 존재하지만 호출 규약이 비호환인 영구 장애를 무기한 일시 실패로 처리한다 (Major)
- **관점**: 2. 숨은 가정, 8. 장애 대응, 9. 운영 복잡도
- **지적**: unsupported 판정을 `search`와 `select` 메서드의 존재 여부로만 제한하면, 메서드는 존재하지만 인자·반환 객체·projection 동작이 현재 전략과 호환되지 않는 버전은 모두 일시 실패로 분류된다. 이 경우 복구되지 않는 동일 오류를 30분마다 계속 실행하면서 purge는 성공하지 못한다.
- **근거**: A-0002는 `select(...).to_arrow()` 호출 중의 `TypeError`, `AttributeError` 등을 내부 결함일 수 있다는 이유로 모두 `failed`로 처리한다. 이는 오분류 방지에는 유리하지만, “비호환 버전에서는 메서드 자체가 없다”는 검증되지 않은 전제에 의존한다. 호출 규약이 다른 배포 환경에서는 24시간 reconciliation도 성공하지 못하고, 업데이트 안내용 unsupported UI 알림도 제공되지 않는다.
- **대안**: 배포 의존성을 검증된 LanceDB 버전으로 정확히 고정하고 시작 시 버전·호출 전략 호환성을 fail-fast 검증하는 것이 가장 단순하다. 여러 버전을 지원해야 한다면 동일 `capability_sig`에서 동일한 호출 단계·예외 지문이 일정 횟수 연속 발생할 때만 `unsupported`로 승격하고 UI에 업데이트 안내를 표시하되, DB I/O 등 명백한 일시 오류는 기존 백오프에 유지하는 제한적 회로 차단기를 둘 수 있다.

### [m-1] “숨은 기본 limit 없음”에 대한 현재 실측 근거가 충분하지 않다 (Minor)
- **관점**: 2. 숨은 가정, 4. 확장성, 5. 성능
- **지적**: ADR은 숨은 기본 limit이 없다고 확정하지만, 명시된 완전성 검증은 500/500행뿐이다. 흔한 기본 제한값이 500보다 큰 경우 이 시험으로는 제한 유무를 판별할 수 없다.
- **근거**: 실제 운영 및 필수 성능 시험은 10만 행을 전제로 하고, projection 누락은 제거 대상 경로의 일부를 영구적으로 보존하는 정확성 결함이 된다. 검증 계획의 10만 행 반환 건수 비교는 적절하지만, 그 결과가 아직 확보됐다는 내용은 문서에 없다.
- **대안**: 배포 전 게이트에서 10만 행 전체 반환과 앞·중간·끝 sentinel 포함을 실제로 검증하고 결과를 ADR 근거에 기록한다. 완료 전까지 ADR의 표현은 “0.34.0에서 500행 예비 검증, 10만 행 배포 게이트 예정”으로 제한한다.

### [m-2] dirty marker 삭제 테스트 문언이 1초 bounded join 설계와 불일치한다 (Minor)
- **관점**: 8. 장애 대응
- **지적**: 검증 계획은 `clear_dirty_shutdown()`의 “비동기 즉시 반환”을 테스트한다고 되어 있지만, 최신 실패 대응 설계는 daemon 스레드 시작 후 최대 1초간 `join()`하도록 변경됐다.
- **근거**: A-0001 실패 모드는 `_CLEAR_DIRTY_JOIN_TIMEOUT_SEC=1`의 bounded wait를 명시하는 반면, 검증 계획은 이전 리뷰 단계의 즉시 반환 계약을 유지한다. 구현이 최신 설계를 따르면 즉시 반환 테스트가 실패하거나, 느슨한 테스트라면 1초 상한 회귀를 잡지 못한다.
- **대안**: 테스트 계약을 “정상 unlink는 완료를 관측하고, 블록되는 unlink는 설정된 상한 이내에 반환하며 종료를 계속한다”로 변경한다. 시간 기반 테스트의 흔들림을 줄이려면 join 함수나 timeout을 주입해 전달값과 후속 진행을 검증한다.
