## 총평
두 설계 모두 관측된 문제의 원인을 직접 겨냥하며, 정상 경로뿐 아니라 실패·재실행·운영 상황까지 상당히 구체적으로 다룬 점이 좋다. 특히 종료 경로의 단일 수렴점과 purge projection, 보수적 메타 복구, 24시간 강제 reconciliation은 요구사항과 대체로 잘 정렬된다. 다만 A-0001의 워커 잔존 처리 방식이 ADR-0001과 일치하지 않고, `isRunning()` 자체가 실패한 경우 FR-1을 어떻게 보장할지 결정되지 않았다. 이 두 항목을 명문화하면 큰 재설계 없이 진행할 수 있다.

**판정**: APPROVE_WITH_CHANGES

## 동의하는 결정
- `quitOnLastWindowClosed(False)`와 `_shutdown()`의 명시적 `quit()`을 조합한 것은 트레이 앱의 창 가시성과 프로세스 수명을 분리하는 적절한 결정이다. 종료 로직을 `closeEvent → _shutdown()`으로 수렴시킨 것도 중복 정리와 경로별 동작 차이를 줄인다.
- 정상 종료보다 최종적인 프로세스 종료를 우선하되, 먼저 스케줄러 정지와 워커의 graceful 종료를 시도하는 단계적 정책은 R-0001의 FR-1과 NFR-1 사이의 현실적인 절충이다.
- purge에서 `file_path`만 projection하고 Arrow를 직접 순회하는 방향은 불필요한 벡터·암호문 로드를 원인 단계에서 제거하므로 타당하다. 전체 로드 폴백을 두지 않고 실제 컬럼 pushdown을 채택 전제조건으로 둔 것도 “결과에서만 버리는 가짜 projection”을 방지한다.
- purge의 성공 스킵에 24시간 상한을 둔 것은 유휴 비용을 줄이면서 외부 요인에 의한 state-DB 불일치를 무기한 방치하지 않는 합리적인 절충이다.
- 실패·차단 판정을 성공 스킵보다 먼저 수행하고, 일시적 실패 시 `reconciled_sig`를 해제하는 상태 전이는 백오프 만료 후 재시도가 성공 메타에 가로막히는 문제를 적절히 방지한다.
- op_sig에 정규화된 폴더 목록뿐 아니라 `dry_run`, `max_delete_ratio`, 스키마 버전을 포함하고 canonical JSON으로 해시하는 결정은 프로세스 재시작과 설정 변경에도 안정적인 판정을 제공한다.
- sidecar 손상·부재를 “스킵 불가”로 취급하는 보수적 복구 정책과 tmp→replace 저장은 장애 시 삭제 누락보다 재실행을 선택하므로 안전하다.

## 지적 사항

### [M-1] ADR-0001과 현재 종료 설계의 결정 범위가 일치하지 않음 (Major)
- **관점**: 1. 요구사항 누락, 3. 아키텍처 결함
- **지적**: ADR-0001은 `_shutdown()` 마지막에 `QApplication.instance().quit()`을 예외와 무관하게 호출하고 기존 `stop_worker` 에스컬레이션은 그대로 유지한다고 결정한다. 반면 A-0001은 `stop_worker()` 예외 후 워커가 실행 중이면 `quit()` 대신 별도의 `hard_exit`를 호출하는 추가 종료 정책을 도입한다.
- **근거**: A-0001의 “단계별 실패 정책”과 검증 계획에는 `quit` 또는 `hard_exit` 중 정확히 하나를 호출한다고 되어 있지만, 이 분기와 `_shutdown()` 차원의 추가 hard-exit 책임은 ADR-0001의 결정·결과에 기록되어 있지 않다. 따라서 ADR을 구현 기준으로 읽으면 무조건 `quit()`, architecture를 기준으로 읽으면 조건부 `hard_exit()`가 되어 결정 이력이 이원화된다.
- **대안**: ADR-0001의 결정을 “정리 후 워커 비실행이 확인되면 `quit()`, 워커 실행 또는 안전한 판정 불가 시 `hard_exit()`”로 갱신하고 대안·트레이드오프를 기록한다. 또는 `_shutdown()`의 추가 분기를 제거하고 `stop_worker()`가 모든 잔존 판정과 hard-exit를 책임지도록 계약을 강화해 ADR의 “기존 에스컬레이션 유지”와 일치시킨다.

### [M-2] `worker.isRunning()` 조회 실패 시 종료 보장 정책이 정의되지 않음 (Major)
- **관점**: 7. 동시성, 8. 장애 대응
- **지적**: 워커 실행 여부 조회 자체가 예외를 던졌을 때 `quit()`과 `hard_exit()` 중 무엇을 선택할지 명확하지 않다. 실행 상태를 알 수 없는 상황에서 단순 `quit()`을 선택하면 QThread 잔존으로 FR-1의 프로세스 종료 보장이 깨질 수 있다.
- **근거**: 검증 계획은 `isRunning` 예외도 매개변수화한다고 명시하면서 결과를 “quit(워커 미실행) 또는 hard_exit(워커 잔존)”라고 적었다. 그러나 `isRunning()`이 실패하면 해당 두 상태를 판별할 관측값 자체가 없다.
- **대안**: 종료 우선 정책에 맞춰 실행 상태가 “실행 중 또는 판정 불가”이면 보수적으로 `hard_exit()`하도록 명시한다. 더 단순하게는 `stop_worker()`가 `STOPPED`, `FORCED_STOPPED`, `UNKNOWN` 같은 결과를 반환하게 하고 `_shutdown()`은 `STOPPED`에서만 `quit()`, 나머지는 hard-exit하도록 계약화할 수 있다.

### [m-1] purge 성공 후 sidecar 저장 실패 시 메모리 상태의 커밋 규칙이 불명확함 (Minor)
- **관점**: 7. 동시성, 8. 장애 대응
- **지적**: 문서는 실패·차단 상태는 저장 성공 여부와 무관하게 메모리에 즉시 반영한다고 명시하지만, purge 성공 메타가 sidecar 저장 실패 시 메모리에 유지되는지는 정의하지 않는다.
- **근거**: 실패 모드에는 “purge 성공 후 meta 저장 실패 → 다음 사이클 재실행”이라고 되어 있다. 그러나 성공 메타를 먼저 메모리에 반영하고 파일 저장만 실패하면 같은 프로세스의 다음 사이클은 `reconciled_sig`와 `last_purge_ts`를 보고 24시간 스킵할 수 있어 문서의 재실행 설명과 충돌한다.
- **대안**: 성공 상태는 sidecar replace 성공 후에만 메모리의 committed meta로 승격하거나, 저장 실패 시 성공 필드를 메모리에서도 롤백한다고 명시한다. 반대로 메모리 성공 상태를 유지하려면 실패 모드 설명을 “현 프로세스에서는 스킵 가능하나 재시작 후 재실행”으로 고치고 그 동작을 테스트한다.

### [m-2] R-0002의 10만 청크 메모리 상한에 대한 검증 계획이 부족함 (Minor)
- **관점**: 4. 확장성, 5. 성능
- **지적**: 요구사항은 청크 10만 기준 추가 할당을 수십 MB 이내로 실측하도록 규정하지만, 현재 검증 계획은 유휴 1시간 RSS 추이와 projection pushdown 확인만 명시한다.
- **근거**: RSS 평탄화는 반복 누적 문제를 보여주지만, 10만 청크 강제 reconciliation 한 번의 peak allocation이 NFR-1 상한을 만족하는지는 별도로 입증하지 못한다.
- **대안**: 10만 개 수준의 대표 길이 `file_path` 데이터를 가진 테스트 DB에서 강제 purge를 실행하고 프로세스 peak RSS 또는 Arrow memory pool 최대치를 전후 비교하는 성능 수용 테스트를 추가한다. 기준은 환경 편차를 고려해 “baseline 대비 추가 할당 수십 MB 이내”처럼 측정 방법과 함께 고정한다.

### [m-3] R-0001의 시간 제한이 검증 계획에 명시적으로 연결되지 않음 (Minor)
- **관점**: 1. 요구사항 누락, 8. 장애 대응
- **지적**: 검증 계획은 프로세스 소멸 여부를 확인하지만 유휴 3초, 행오버 15초라는 수용 기준을 실제로 측정한다고 명시하지 않는다.
- **근거**: R-0001 AC-1~AC-3에는 구체적인 종료 시간이 있으나 A-0001의 실기 항목은 “프로세스 소멸 확인”, “에스컬레이션 동작 확인”에 그친다.
- **대안**: 종료 명령 시점부터 부모 프로세스 소멸까지 elapsed time을 측정해 각 경로에서 3초 또는 15초 이내인지 판정하도록 릴리스 체크리스트나 subprocess 기반 통합 테스트에 추가한다.

### [N-1] ADR-0002의 projection 미지원 정책과 요구사항 문구 정렬 (Nit)
- **관점**: 1. 요구사항 누락
- **지적**: R-0002 FR-1은 projection이 불가능하면 “최적화 비활성을 경고 로그로 명시”한다고 표현하지만, ADR-0002와 A-0002는 변경 자체를 배포하지 않는다고 표현한다.
- **근거**: 두 정책 모두 조용한 전체 로드 폴백을 금지한다는 핵심은 같지만, 런타임에 기능을 비활성화하는 것인지 빌드·배포를 차단하는 것인지 운영 동작이 다르게 읽힐 수 있다.
- **대안**: 요구사항을 “지원 여부는 빌드·배포 전에 검증하며, 미지원 버전에서는 변경을 배포하지 않고 경고를 남긴다”로 통일하거나, 실제 런타임 비활성 모드를 지원할 계획이면 architecture에 해당 동작과 purge 기능 보존 방식을 추가한다.

## 확인 필요
- 전달된 소스 코드가 없어 `closeEvent`의 모든 종료 경로 수렴, `stop_worker()`의 실제 반환·예외 계약, 워커 종료 전 state 저장, 보조 스레드의 daemon 속성은 확인하지 못했다. 구현 리뷰에서 해당 코드와 회귀 테스트를 확인해야 한다.
- 배포 고정 LanceDB 버전에서 `table.to_lance().to_table(columns=["file_path"])` 계열이 공개·안정 API인지, 그리고 실제 스캔 단계에서 vector/text를 읽지 않는지는 확인 필요하다. 실행 계획대로 스캔 컬럼 또는 메모리 프로파일로 검증해야 한다.
- `max_delete_ratio`, 강제 주기, 백오프 값에 대해 NaN·무한대·음수·과도한 값이 설정 계층에서 차단되는지 확인 필요하다. 차단되지 않는다면 op_sig 생성과 시간·비율 판정 전에 유한성 및 범위 검증이 필요하다.
