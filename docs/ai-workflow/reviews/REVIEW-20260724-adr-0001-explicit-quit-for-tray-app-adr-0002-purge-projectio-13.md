## 총평
두 ADR 모두 문제의 직접 원인을 정확히 겨냥하며, 종료 경로 수렴과 purge projection·보수적 메타 전이는 전반적으로 타당하다. 특히 예외 시에도 종료를 완수하는 최종 분기와, 실패·차단 억제를 성공 스킵보다 먼저 판정하는 결정은 잘 설계됐다. 다만 dirty-shutdown 표식을 `quit()` 요청 전에 지우는 현재 계약은 실제 정상 종료가 확정되기 전에 증거를 없애는 결함이 있고, 단일 인스턴스 획득과 표식 처리의 순서도 본문에 보장되지 않는다. 또한 A-0002의 필수 메모리 수용 기준은 누적 peak RSS를 반복 측정하는 방식이라 거짓 합격할 수 있으므로 수정이 필요하다.

**판정**: REQUEST_CHANGES

## 동의하는 결정
- `quitOnLastWindowClosed(False)`와 `_shutdown()`의 명시적 `quit()`을 함께 적용한 것은 트레이 앱의 창 가시성과 프로세스 수명을 분리하는 적절한 결정이다.
- `stop_worker()` 예외 이후에도 `isRunning()`을 재판정하고, 실행 중이거나 판정 불가하면 `hard_exit`로 수렴시키는 정책은 “종료가 반드시 완료된다”는 우선 요구와 일치한다.
- 하드 종료 경로에서 동기 파일 I/O와 `logging.shutdown()`을 제거한 것은 잠금·백신·파일시스템 지연 때문에 최후 안전망 자체가 멈추는 역설을 피한다.
- purge에서 `file_path`만 projection하고, 성공·실패·차단 상태를 sidecar 저장 성공 여부와 무관하게 메모리에 즉시 반영하는 설계는 성능과 재실행 안전성을 균형 있게 다룬다.
- 실패 시 `reconciled_sig`를 해제하고 판정 순서를 차단 → 백오프 → 성공 스킵 → 실행으로 고정한 것은 이전 성공 상태가 재시도를 가리는 상태 전이 결함을 방지한다.
- 메타 손상·유실 시 스킵하지 않고 멱등 purge를 재실행하며, 비정상 삭제 비율은 fail-closed로 처리하는 방향에 동의한다.

## 지적 사항

### [M-1] dirty-shutdown 표식을 정상 종료 확정 전에 삭제함 (Major)
- **관점**: 7. 동시성, 8. 장애 대응
- **지적**: `clear_dirty_shutdown()`을 `quit_fn()`보다 먼저 호출하는 계약은 “정상 quit 확정 시에만 표식을 지운다”는 설명과 일치하지 않는다. `quit()`은 이벤트 루프 종료를 요청할 뿐 프로세스의 정상 종료 완료를 보장하지 않는다.
- **근거**: A-0001 검증 계획은 “quit 경로에서만 `clear_dirty()`가 `quit_fn()`보다 먼저 호출되는지”를 고정한다. 표식을 지운 뒤 `quit_fn()`이 예외를 던지거나, 이벤트 루프 반환 전 네이티브 크래시·강제 종료가 발생하면 다음 시작에서 dirty 상태를 탐지하지 못하는 false negative가 생긴다.
- **대안**:
  1. `_shutdown()`에서는 `quit()`만 요청하고, `app.exec()`가 정상 반환한 뒤 `main()`의 정상 반환 경로에서 표식을 삭제한다.
  2. 시작 시 표식 생성 → 이벤트 루프 실행 → `app.exec()` 정상 반환 후 표식 삭제를 `try/finally`가 아닌 명시적 성공 경로로 구성한다. 예외 및 hard-exit 경로에서는 삭제하지 않는다.
  3. 테스트도 “clear가 quit보다 먼저”가 아니라 “정상적인 `app.exec()` 반환 후에만 clear, 예외·hard-exit·비정상 반환 전에는 유지”로 변경한다.

### [M-2] 단일 인스턴스 획득과 dirty marker 처리 순서가 설계 본문에 보장되지 않음 (Major)
- **관점**: 2. 숨은 가정, 7. 동시성, 8. 장애 대응
- **지적**: dirty marker 확인·재기록은 반드시 단일 인스턴스 소유권을 획득한 프로세스에서만 수행되어야 하지만, 리뷰 대상 A-0001에는 이 순서와 책임이 명시돼 있지 않다.
- **근거**: 실행 중인 정상 주 인스턴스는 설계상 marker를 계속 보유한다. 이때 두 번째 실행 프로세스가 단일 인스턴스 판정보다 먼저 `check_and_remark_dirty_shutdown()`을 호출하면, 정상 실행 중인 주 인스턴스의 marker를 “직전 비정상 종료”로 오판해 경고할 수 있다. ADR 헤더에는 “단일인스턴스 순서 정정”이 언급되지만, 현재 아키텍처 본문의 컴포넌트·데이터 흐름·검증 계획에는 그 불변식이 나타나지 않아 향후 리팩터링에서 퇴행할 수 있다.
- **대안**:
  1. 시작 흐름을 `QApplication 생성 → 단일 인스턴스 획득/기존 인스턴스 통지 → 소유권 획득 성공 시에만 dirty marker 확인·재기록 → MainWindow 생성 → app.exec()`로 명문화한다.
  2. 기존 인스턴스를 발견한 보조 프로세스는 marker를 읽거나 쓰거나 지우지 않는다는 불변식을 추가한다.
  3. 주 인스턴스 실행 중 두 번째 실행을 시뮬레이션해 dirty 알림과 marker 변경이 없음을 검증하는 회귀 테스트를 추가한다.

### [M-3] peak RSS 반복 측정 방식이 메모리 수용 기준을 검증하지 못함 (Major)
- **관점**: 5. 성능, 8. 장애 대응
- **지적**: `resource.getrusage(...).ru_maxrss` 같은 프로세스 수명 누적 peak를 각 반복의 직전·직후에 읽고 차이를 낸 뒤 5회 중앙값을 구하면, 첫 실행 이후 값이 0에 가까워져 실제 추가 할당량이 30MB를 초과해도 합격할 수 있다.
- **근거**: A-0002의 필수 성능 수용 절차는 동일 프로세스에서 warm-up 후 5회 반복하고 “직후 peak RSS - 직전 peak RSS”를 사용한다. `ru_maxrss`는 현재 RSS가 아니라 과거 최고값이므로 첫 측정에서 최고점을 갱신한 뒤 동일하거나 더 낮은 메모리를 사용하는 후속 4회는 차이가 0이 된다. 중앙값은 0이 되어 구조적으로 거짓 합격이 가능하며, Windows에서는 해당 API의 가용성과 단위도 측정 환경에 따라 다르다.
- **대안**:
  1. 각 반복을 새로운 자식 프로세스에서 실행하고, 시작 안정 RSS 대비 실행 중 샘플링한 최대 RSS의 차이를 측정한다.
  2. Windows 운영 환경에서는 `psutil.Process().memory_info().rss` 또는 동등한 OS API를 짧은 간격으로 샘플링해 `max(sampled_rss) - baseline_rss`를 계산한다.
  3. 5개 독립 프로세스 측정값의 중앙값과 최댓값을 함께 기록하고, 최소한 중앙값 30MB 이하 및 명시적 단회 상한을 둔다.
  4. NFR이 purge 전체 메모리에 관한 것이라면 projection 쿼리뿐 아니라 Arrow 순회, 삭제 대상 구성 및 `optimize()`까지 포함한 end-to-end 강제 reconciliation을 측정한다.

### [m-1] projection 전건 반환 검증의 행 수 기준이 부족함 (Minor)
- **관점**: 4. 확장성, 5. 성능
- **지적**: 500/500행 확인만으로 모든 규모에서 숨은 limit이 없다고 단정하기보다는, 필수 10만 행 성능 시험에서 반환 행 수와 purge 대상 판정의 완전성도 함께 확인할 필요가 있다.
- **근거**: purge는 일부 행만 반환돼도 예외 없이 “성공”할 수 있으며, 그러면 `reconciled_sig`가 기록되어 누락된 고아 경로의 재검사가 최대 24시간 지연된다. 현재 필수 성능 절차에는 결과 행 수 검증이 명시돼 있지 않다.
- **대안**: 10만 행 DB에서 projection 반환 행 수가 실제 테이블 행 수와 같은지 검증하고, 앞·중간·끝 위치에 배치한 제거 대상 경로가 모두 삭제 후보로 분류되는 회귀 테스트를 추가한다.
