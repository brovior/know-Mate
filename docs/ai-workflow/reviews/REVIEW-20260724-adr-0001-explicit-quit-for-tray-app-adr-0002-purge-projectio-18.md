## 총평
두 ADR 모두 문제 원인을 구체적으로 식별하고, 정상 경로뿐 아니라 예외·강제 종료·메타 저장 실패까지 상태 전이를 설계한 점이 좋다. A-0001의 명시적 `quit()` 전환과 A-0002의 projection 도입은 각각 프로세스 잔존과 반복적인 대용량 로드의 직접 원인에 대응하는 타당한 결정이다. 다만 A-0002의 필수 성능 시험은 projection 단계를 먼저 실행한 동일 프로세스에서 end-to-end 단계를 다시 측정하므로, 캐시와 이미 확장된 Python/Arrow 메모리 풀 때문에 최종 합격 지표가 실제 최초 purge 비용을 과소평가할 수 있다. 구현 소스와 R-0001/R-0002 요구사항 원문은 제공되지 않아 문서에 기재된 구현 완료 및 요구 충족 여부 자체는 검증할 수 없다.

**판정**: REQUEST_CHANGES

## 동의하는 결정
- A-0001에서 `quitOnLastWindowClosed(False)`와 모든 종료 경로의 `_shutdown()` 수렴을 함께 적용한 결정에 동의한다. 트레이 앱의 창 가시성과 프로세스 수명을 분리해 종료 동작을 명시적으로 통제하는 구조다.
- `QThread.terminate()` 사용 여부를 별도 반환하여 이후 `isRunning()==False`만으로 정상 종료를 판정하지 않는 결정은 적절하다. 강제 중단 시 락과 저장 상태를 신뢰할 수 없다는 근거가 명확하다.
- hard-exit 경로에서 동기 로그 flush나 파일 표식 I/O를 제거한 결정에 동의한다. 최후 안전망 자체가 락이나 파일 I/O에 막히는 역설을 방지한다.
- A-0002에서 전체 테이블 로드 대신 `file_path` projection을 사용하고, API 부재만 `"unsupported"`로 좁게 분류한 결정은 타당하다. 내부 오류를 영구 비호환으로 오분류하는 위험도 적절히 고려했다.
- purge 메타를 기존 state와 분리하고, 저장 실패와 무관하게 프로세스 내 상태를 먼저 전이하는 설계는 기존 소비자 호환성과 반복 부하 억제를 함께 만족한다.
- 차단·백오프·성공 스킵·실행의 판정 순서와 실패 시 `reconciled_sig` 해제는 재시도가 성공 메타에 가려지는 문제를 방지하는 일관된 상태 모델이다.

## 지적 사항

### [M-1] end-to-end 성능 시험이 선행 projection으로 워밍업된다 (Major)
- **관점**: 5. 성능, 8. 장애 대응
- **지적**: 필수 성능 시험에서 같은 프로세스가 먼저 “projection + Arrow 순회 단계”를 실행한 뒤, 다시 projection을 포함한 end-to-end 단계를 실행한다. 따라서 최종 NFR 판정 대상인 두 번째 단계는 실제 최초 purge와 동일한 cold 상태가 아니다.
- **근거**: A-0002 검증 계획은 warm-up에 projection을 포함하지 말아야 한다고 정확히 설명하지만, 바로 다음 절차에서 단계 1이 projection을 수행하고 단계 2가 다시 `1 + 소속 판정 + 삭제 대상 구성 + optimize()`를 수행한다. 단계 1에서 OS 파일 캐시, LanceDB 내부 캐시, PyArrow 메모리 풀 및 CPython 힙이 이미 준비될 수 있다. 해제된 메모리가 RSS에 남는 환경에서는 두 번째 projection이 추가 RSS를 거의 만들지 않아 `max(sampled_rss)-baseline_rss`가 실제 신규 프로세스의 end-to-end 비용보다 작게 측정될 수 있다. 이는 문서가 앞서 배제한 “측정 대상 자체를 warm-up하여 거짓 합격” 문제를 단계 분리 방식으로 다시 도입한다.
- **대안**:
  1. 최종 판정용 end-to-end를 각 독립 프로세스에서 **가장 먼저 한 번만** 수행하고, projection 단독 진단은 별도의 프로세스와 별도 DB 복제본에서 측정한다.
  2. 5개 end-to-end 프로세스와 5개 projection 진단 프로세스를 분리하고, NFR 판정은 전자만 사용한다.
  3. 프로세스 수를 늘리기 어렵다면 end-to-end를 먼저 측정한 뒤 projection 단독 측정은 원인 분석용 참고값으로만 수집한다고 순서를 바꾼다.
  4. Windows `PeakWorkingSetSize`는 프로세스 수명 누적 피크이므로, 각 판정 프로세스가 측정 대상 하나만 수행하도록 해 단계 귀속을 명확히 한다.

### [m-1] 성능 시험의 삭제·optimize 실행 조건이 고정되지 않았다 (Minor)
- **관점**: 2. 숨은 가정, 5. 성능
- **지적**: end-to-end 시험이 실제 삭제 및 `optimize()`를 실행하도록 하는 설정값과 사전조건이 명시되지 않았다.
- **근거**: 문서는 end-to-end 범위를 `소속 판정 + 삭제 대상 구성 + optimize()`까지로 정의하지만, 테스트 설정의 `dry_run`, `max_delete_ratio`, watch folder 구성, 삭제 후보 비율을 고정하지 않는다. 기본 `dry_run=true`이거나 후보 비율이 차단율을 넘으면 실제 삭제 또는 optimize 경로를 타지 않은 채 합격할 수 있다.
- **대안**: 시험 명세에 `dry_run=false`, 유효한 고정 `max_delete_ratio`, 삭제 후보 비율이 차단 임계 이하임을 명시하고, 실행 후 실제 삭제 건수와 `optimize()` 호출 여부를 assertion 또는 스파이/로그로 검증한다. 대량삭제 차단 경로는 별도 기능 시험으로 분리한다.

### [m-2] projection 이후에도 purge 메모리는 전체 행 수에 선형 증가한다 (Minor)
- **관점**: 2. 숨은 가정, 4. 확장성
- **지적**: 설계는 벡터와 원문 로드를 제거하지만, 모든 청크의 `file_path`를 Arrow 테이블로 한 번에 적재하므로 purge의 시간·메모리 복잡도는 여전히 전체 청크 수에 대해 O(N)이다. 지원 가능한 최대 행 수에 대한 가정이 문서화되지 않았다.
- **근거**: 검증 게이트는 10만 행만 사용한다. 현재 수만 청크에서는 충분할 수 있으나 데이터가 10배 이상 증가하거나 긴 네트워크 경로가 많은 경우 경로 컬럼 자체와 Arrow 오프셋, Python 삭제 후보 구조가 60MB 상한을 넘을 수 있다. “벡터 크기와 무관”하다는 설명은 맞지만 “인덱스 규모와 무관”한 구조는 아니다.
- **대안**: 10만 행을 제품 지원 상한 또는 대표 용량으로 선택한 근거를 명시하고, 예상 최대 청크 수에서도 시험한다. 장기적으로는 LanceDB가 지원한다면 배치/스트리밍 순회, 서버 측 조건 필터, 파일별 중복 경로 축소 등의 방식을 검토한다.

### [N-1] dirty-shutdown 표식의 의미가 일부 문구와 다르다 (Nit)
- **관점**: 8. 장애 대응
- **지적**: 시작 시 표식을 기록하고 정상 `app.exec()` 반환 후에만 지우는 방식은 hard exit뿐 아니라 크래시, 전원 종료, 시작 후 예외, 삭제 실패도 함께 탐지한다. 따라서 “강제 종료가 있었다는 사실”이라는 표현은 실제 검출 범위보다 좁다.
- **근거**: 최대 1초의 best-effort 삭제가 실패해도 다음 시작에서 표식이 발견될 수 있으며, 이는 강제 종료의 확정 증거가 아니다. 문서도 false positive 가능성을 인정한다.
- **대안**: 명칭과 UI 문구를 “이전 실행이 정상 종료를 완료하지 못했거나 종료 표식을 정리하지 못함”으로 표현하고, 재인덱싱 안내 역시 가능성 기반 문구로 유지한다.

## 확인 필요
- R-0001/R-0002 요구사항 원문이 제공되지 않아 AC 및 NFR과 설계의 완전한 추적성은 확인할 수 없다.

- `table.search().select(["file_path"]).to_arrow()`의 10만 행 이상 전건 반환과 컬럼 pruning은 문서상 실측·후속 시험으로만 제시되어 있으므로, 배포 전 성능 게이트 결과 확인이 필요하다.
