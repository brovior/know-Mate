## 총평
두 설계 모두 문제 원인과 요구사항의 연결이 명확하며, 명시적 종료와 컬럼 projection이라는 핵심 결정도 타당하다. 특히 종료 단계의 보수적 `hard_exit` 분기와 purge의 24시간 강제 reconciliation·실패 백오프는 기존 기능을 훼손하지 않으면서 운영 위험을 줄이는 합리적인 선택이다. 다만 A-0001의 요약 흐름과 실패 모드가 ADR의 조건부 `quit()`/`hard_exit` 결정과 서로 모순되어 구현자가 핵심 안전망을 누락할 수 있다. 또한 종료의 정확히 한 번 실행과 purge 메타 저장 실패 후의 실제 재실행 의미를 조금 더 명확히 해야 한다. 소스 코드는 제공되지 않았으므로 기존 코드에 관한 사실과 LanceDB pushdown은 별도 확인이 필요하다.

**판정**: APPROVE_WITH_CHANGES

## 동의하는 결정
- `quitOnLastWindowClosed(False)`와 모든 종료 경로의 `_shutdown()` 수렴을 함께 적용한 결정에 동의한다. 트레이 앱의 종료를 창 가시성이나 Qt 암묵 규칙과 분리해 예측 가능하게 만든다.
- 워커 비실행이 확인된 경우에만 `quit()`하고, 실행 중이거나 판정 불가하면 `hard_exit`로 전환하는 보수적 최종 분기는 FR-1과 기존 행오버 안전망을 일관되게 연결한다.
- purge 조회를 `file_path` projection과 Arrow 직접 순회로 제한하고, 실제 컬럼 pushdown 확인을 배포 전제조건으로 둔 것은 “결과에서만 컬럼을 버리는 가짜 projection”을 방지하는 적절한 결정이다.
- purge 스킵에 24시간 강제 reconciliation을 결합하고, 실패·차단 억제 판정을 성공 스킵보다 먼저 수행한 것은 복구 기회와 장애 증폭 방지를 균형 있게 다룬다.
- 기존 `index_state.json` 대신 원자 교체되는 sidecar를 사용한 것은 기존 스키마 소비자에 대한 마이그레이션 위험을 줄이는 합리적인 선택이다.

## 지적 사항

### [M-1] A-0001 요약 흐름이 조건부 최종 종료 결정과 모순됨 (Major)
- **관점**: 1. 요구사항 누락, 8. 장애 대응
- **지적**: A-0001의 컴포넌트 표·데이터 흐름·실패 모드에는 `_shutdown()` 마지막에 항상 `QApplication.quit()`을 호출하는 것으로 표현되어 있지만, ADR과 정상 종료 계약은 `worker.isRunning()` 결과에 따라 `quit()` 또는 `hard_exit` 중 하나를 선택하도록 결정하고 있다.
- **근거**: 컴포넌트 표는 “마지막에 `QApplication.instance().quit()`”, 데이터 흐름은 무조건 `QApplication.quit()`, 실패 모드는 “`quit()`은 finally 위치로 보장”이라고 적혀 있다. 반면 ADR-0001과 같은 문서의 정상 종료 계약은 워커 실행 중 또는 조회 예외 시 `hard_exit`를 호출하며 `quit()`과 정확히 하나만 실행한다고 규정한다. 구현자가 앞부분만 따르면 `stop_worker()` 예외 후 실행 중인 QThread를 남긴 채 `quit()`하여 FR-1을 다시 위반할 수 있다.
- **대안**: 컴포넌트 표와 데이터 흐름의 마지막 단계를 `finalize_shutdown(worker)` 같은 조건부 판정으로 통일하고, 실패 모드도 “최종 판정은 항상 도달하며 결과로 quit 또는 hard_exit 중 하나를 실행”으로 수정한다. 의사코드와 테스트 이름도 동일한 계약을 사용한다.

### [m-1] `_shutdown()`의 프로세스 수명 기준 멱등성 계약이 명시되지 않음 (Minor)
- **관점**: 7. 동시성
- **지적**: 문서는 `quit()`과 `hard_exit`가 정확히 하나만 실행된다고 하지만, 이는 한 번의 `_shutdown()` 호출 내부인지 프로세스 수명 전체인지 명확하지 않으며 중복 close 이벤트나 종료 시그널에 대한 가드가 설계되어 있지 않다.
- **근거**: 종료 경로는 모두 `closeEvent → _shutdown()`으로 수렴하지만, 트레이 종료와 X 종료가 근접하게 발생하거나 향후 종료 경로가 추가되면 `_shutdown()`이 두 번 진입해 스케줄러·트레이·워커 정리를 반복할 수 있다. 현재 UI 스레드에서 순차 처리될 가능성이 높아 위험은 제한적이지만, 문서가 주장하는 “정확히 하나”를 회귀 테스트로 보장하기 어렵다.
- **대안**: `_shutdown_started`/`_shutdown_finished` 상태를 두거나 단일 `request_shutdown()` 진입점을 두어 중복 호출을 멱등 처리한다. 두 종료 요청을 연속 발생시켜 `stop_worker`와 최종 종료 함수가 한 번만 실행되는 테스트를 추가한다.

### [m-2] purge 성공 후 sidecar 저장 실패 시 “다음 사이클 재실행” 보장이 성립하지 않을 수 있음 (Minor)
- **관점**: 7. 동시성, 8. 장애 대응
- **지적**: A-0002는 성공 메타 저장 실패 시 메모리에도 신규 성공 상태를 반영하지 않으므로 다음 사이클에 재실행한다고 설명하지만, 메모리에 이전의 유효한 `reconciled_sig`와 24시간 미경과 `last_purge_ts`가 남아 있으면 다음 0건 사이클은 성공 스킵될 수 있다.
- **근거**: 처리 건이 있어 purge를 실행했으나 sidecar replace가 실패한 경우를 가정하면, 이전 메타가 같은 `op_sig`의 최근 성공 기록일 수 있다. 이때 다음 사이클은 `op_sig == reconciled_sig && 처리 0건 && 24h 미경과`가 성립하여 재실행되지 않는다. purge 자체는 이미 성공했으므로 직접적인 데이터 오류는 아니지만, 문서의 실패 복구 설명과 예상 테스트가 실제 상태 전이와 불일치한다.
- **대안**: 다음 중 하나를 명시적으로 선택한다.  
  1. DB purge 성공 상태는 메모리에는 승격하고 디스크에만 미반영하여 현재 프로세스의 반복 조회를 막고, 재시작 시 보수적으로 재실행한다.  
  2. 반드시 다음 사이클 재실행이 필요하다면 저장 실패 시 메모리의 `reconciled_sig`를 무효화하되, sidecar 장기 쓰기 실패가 매분 O(N) 조회를 유발하지 않도록 별도 메타 저장 재시도 백오프를 둔다.  
  3. 현재 동작을 유지하려면 “다음 사이클에 재실행될 수 있으며, 늦어도 기존 강제 reconciliation 시점에 다시 실행”으로 문언과 테스트를 수정한다.

### [m-3] `last_purge_ts`의 미래 시각 허용 정책이 문서 내에서 일관되지 않음 (Minor)
- **관점**: 2. 숨은 가정, 8. 장애 대응
- **지적**: ADR, A-0002 데이터 흐름, 실패 모드가 미래 `last_purge_ts`를 서로 다르게 취급한다.
- **근거**: ADR-0002는 미래값이면 무효라고 규정하고, A-0002 주석은 `now + 오차허용`보다 큰 경우에만 무효라고 표현한다. 그러나 실제 스킵 조건은 `0 <= now - last_purge_ts`여서 오차허용 범위 내의 작은 미래값도 스킵 불가이며, 실패 모드 표 역시 `now < last_purge_ts`이면 무시한다고 적혀 있다.
- **대안**: 정책을 하나로 통일한다. 가장 단순한 선택은 모든 미래값을 무효 처리하고 조건·표·ADR에서 오차허용 표현을 제거하는 것이다. 시계 오차를 허용하려면 `-clock_skew_tolerance <= now-last_purge_ts < interval`처럼 판정식을 명시하고 해당 경계 테스트를 추가한다.

### [N-1] A-0002 개요의 스킵 조건이 최종 결정 전체를 반영하지 않음 (Nit)
- **관점**: 1. 요구사항 누락
- **지적**: 개요에는 “watch_folders 구성이 동일하고 처리 건수가 0이면 스킵”이라고만 되어 있어 `dry_run`, `max_delete_ratio`, 24시간 상한, 실패·차단 선행 판정이 생략되어 있다.
- **근거**: 본문의 최종 조건은 단순 폴더 구성 비교가 아니라 `op_sig`와 시간·억제 상태를 포함한다. 개요만 읽는 구현자나 운영자가 실제 동작을 오해할 수 있다.
- **대안**: 개요를 “동일 op_sig·처리 0건·마지막 성공 후 24시간 미만이며 실패/차단 억제 상태가 아닌 경우 스킵”으로 갱신한다.

## 확인 필요
- 소스 코드가 제공되지 않았으므로 현재 종료 경로가 실제로 트레이 [종료]와 `close_action=quit` 두 개뿐인지, 모든 경로가 `closeEvent → _shutdown()`으로 수렴하는지 확인이 필요하다.
- `scheduler.stop()`, `stop_worker()`, `worker.isRunning()` 및 최종 종료 판정이 실제 코드에서 단계별 독립 예외 처리로 구현 가능한지 확인이 필요하다.
- 배포 고정 LanceDB 버전에서 `table.to_lance().to_table(columns=["file_path"])`가 공개·안정 API인지, 실제 스캔 단계에서 vector/text 컬럼을 읽지 않는지 실행 계획 또는 메모리 계측으로 확인해야 한다.
- `file_path` 기준 삭제와 `optimize()` 재실행이 현재 LanceDB 사용 방식에서 논리적으로 멱등한지 기존 삭제 구현을 통해 확인해야 한다.
