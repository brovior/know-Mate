## 총평
두 ADR 모두 문제의 직접 원인을 식별하고 종료 경로 수렴, 컬럼 projection, 주기적 reconciliation 등 합리적인 해결 방향을 선택했다. 특히 실패·차단·성공 스킵의 판정 순서와 메타 상태 전이를 명시한 점은 재실행 안전성을 상당히 높인다. 다만 강제 종료 직전에 `logging.shutdown()`을 호출하는 현재 계약은 잠금 교착으로 인해 “반드시 종료”라는 최상위 요구를 다시 깨뜨릴 수 있다. 또한 A-0002의 핵심 projection 방식이 Accepted 상태임에도 아직 검토·검증 전제로 남아 있고, 강제 종료 중 LanceDB 무결성 보장도 확정되지 않았다.

## 동의하는 결정
- `quitOnLastWindowClosed(False)`와 `_shutdown()`의 명시적 최종 종료를 결합한 것은 트레이 앱의 창 가시성과 프로세스 수명을 분리하는 적절한 결정이다. 모든 정상 종료 경로를 하나의 수렴점으로 모은 것도 중복 정리와 경로별 동작 차이를 줄인다.
- 워커 실행 여부를 확인할 수 없을 때 정상 `quit()`보다 강제 종료를 선택하는 보수적 판정은 “종료 확실성 우선” 요구와 일치한다.
- purge에서 `file_path`만 projection하고 Arrow 컬럼을 직접 순회하는 방향은 벡터·암호문 전체 로드라는 실제 병목을 직접 제거한다.
- 성공 스킵보다 차단·백오프 판정을 먼저 수행하고, 실패 시 `reconciled_sig`를 해제하는 상태 전이는 이전 성공 메타가 재시도를 가리는 문제를 올바르게 방지한다.
- sidecar 손상·유실 시 스킵하지 않고 멱등 purge를 재실행하는 보수적 복구 정책과 24시간 강제 reconciliation은 운영 복잡도 대비 합리적이다.

## 지적 사항

### [B-1] 강제 종료 전 로그 종료가 프로세스 종료를 무기한 막을 수 있음 (Blocker)
- **관점**: 7. 동시성, 8. 장애 대응
- **지적**: `_default_hard_exit`가 `logging.shutdown()` 후 `os._exit()`을 호출하는 계약은 하드 종료의 무조건성을 보장하지 못한다. `logging.shutdown()` 자체가 블로킹될 수 있으므로 `os._exit()`까지 도달하지 못할 가능성이 있다.
- **근거**: A-0001은 `QThread.terminate()` 이후에도 워커가 잔존하거나 상태 판정이 불가능한 상황을 하드 종료 대상으로 삼는다. Python logging 종료는 각 handler의 잠금을 획득하고 flush/close를 수행할 수 있는데, 워커가 로그 handler 잠금을 보유한 채 행오버되었거나 `QThread.terminate()`로 잠금 보유 중 강제 중단되면 메인 스레드의 `logging.shutdown()`이 영구 대기할 수 있다. 이는 R-0001 FR-1을 최상위로 두고 “종료는 반드시 된다”고 한 설계의 핵심 불변식을 직접 위반한다.
- **대안**:
  1. 하드 종료 경로에서는 어떠한 Python 정리 코드도 실행하지 말고 즉시 `os._exit()`을 호출한다. 로그 flush는 graceful 종료 경로에서만 보장한다.
  2. 마지막 로그를 남겨야 한다면 강제 종료 전에 일반 logging이 아닌 비블로킹 또는 OS 수준의 최소 기록을 시도하되, 실패 여부와 무관하게 즉시 `os._exit()`으로 진행한다.
  3. 더 강한 시간 상한이 필요하면 별도 watchdog이 지정 시간 후 현재 프로세스를 `TerminateProcess`하도록 구성한다. 다만 현재 규모에서는 첫 번째 대안이 가장 단순하다.
  4. 회귀 테스트에 “handler 잠금이 다른 스레드에 의해 영구 점유된 상태에서도 hard-exit 함수가 logging 종료를 기다리지 않는다”는 계약을 추가한다.

### [M-1] Accepted 설계에서 핵심 projection API가 아직 미확정 상태임 (Major)
- **관점**: 1. 요구사항 누락, 5. 성능, 8. 장애 대응
- **지적**: A-0002와 ADR-0002가 Accepted이고 구현 완료로 기록되어 있지만, 핵심 성능 수단인 projection API는 여전히 “구현 착수 시 검증”, “1순위로 검토”, “동작 확인 후 확정”으로 기술되어 있다.
- **근거**: ADR-0002는 공개 projection API와 컬럼 pushdown을 채택 전제조건으로 두고 미지원이면 배포하지 않는다고 한다. architecture.md 역시 `table.to_lance().to_table(columns=["file_path"])` 계열을 검토한다고만 명시한다. 어느 API를 실제 채택했는지, 배포 고정 버전이 무엇인지, 해당 API가 정말 vector/text를 읽지 않는지에 대한 확정 결과가 리뷰 대상 문서에는 없다. 이 상태에서는 projection이 단순 결과 컬럼 제거인지 저장소 수준 pushdown인지 판단할 수 없어 핵심 NFR 달성 여부가 미확정이다.
- **대안**:
  1. 검증된 실제 호출 형태와 LanceDB 고정 버전을 ADR 및 architecture.md에 확정 기록한다.
  2. 결과 스키마에 `file_path`만 존재한다는 테스트뿐 아니라, 대표 데이터에서 vector/text 미로드를 Arrow memory pool, peak RSS 또는 LanceDB 실행 관측으로 검증한 결과를 남긴다.
  3. API가 내부/비공개 객체인 `to_lance()`에 의존한다면 업그레이드 안정성을 평가하고, 가능하면 배포 버전의 공개 projection API를 선택한다.
  4. 검증이 아직 수행되지 않았다면 상태를 Accepted가 아닌 Reviewed 또는 조건부 상태로 되돌리고 전제조건 충족 후 다시 승격한다.

### [M-2] LanceDB 쓰기 중 강제 종료의 손상 범위와 복구 계약이 확정되지 않음 (Major)
- **관점**: 6. 보안, 8. 장애 대응
- **지적**: 문서는 강제 종료 시 LanceDB 커밋 원자성을 확인하지 못했다고 인정하면서도, 이어서 “잃는 것은 그 사이클의 state 갱신뿐”이라고 단정한다. 두 서술은 양립하지 않으며, 파생 인덱스 손상 시 감지 및 복구 경로도 A-0001의 실패 모드에 완결되게 정의되어 있지 않다.
- **근거**: A-0001 정상 종료 계약은 `terminate()` 또는 `os._exit()`가 LanceDB add/delete/optimize 도중 발생할 수 있고 영향 범위가 배포 버전의 커밋 원자성에 의존한다고 명시한다. 그 원자성이 미검증이면 다음 실행에서 DB open 실패, 부분 데이터, state와 DB의 불일치가 발생할 가능성을 배제할 수 없다. 데이터 원문은 암호화되어 있고 인덱스는 재생성 가능하므로 회복 불가능한 결함은 아니지만, 베타 사용자가 스스로 복구할 수 있는 감지·격리·재구축 절차가 없으면 재설계 및 운영 비용이 크다.
- **대안**:
  1. 배포 고정 LanceDB 버전에서 add/delete/optimize 각각을 강제 중단한 뒤 DB open·조회·재인덱싱이 가능한지 장애 주입 테스트한다.
  2. 앱 시작 시 최소 DB open/schema/read 검사를 수행하고, 실패 시 손상 인덱스를 격리한 뒤 사용자에게 전체 재인덱싱을 안내하는 복구 경로를 정의한다.
  3. 최소 구현으로 자동 삭제가 부담스럽다면 “인덱스 폴더 백업/삭제 후 전체 재인덱싱” 운영 절차와 UI 또는 로그의 명확한 진단 메시지를 릴리스 문서에 포함한다.
  4. 문서의 “잃는 것은 state 갱신뿐” 표현은 원자성 검증 전까지 제거하고, 가능한 최악의 영향 범위를 일관되게 기술한다.

### [m-1] 문서 최상위 상태와 설계 블록 상태가 불일치함 (Minor)
- **관점**: 9. 운영 복잡도
- **지적**: architecture.md 최상위 상태는 `Draft`, 마지막 갱신은 `(채택일로 갱신)`인 반면 A-0001과 A-0002는 Accepted이며 구현 완료로 기록되어 있다.
- **근거**: 동일 문서 안에서 승인 상태가 상충하면 이후 리뷰어나 릴리스 담당자가 이 문서를 확정 설계로 사용해도 되는지 판단하기 어렵다.
- **대안**: 최상위 상태와 마지막 갱신일을 실제 상태에 맞게 갱신하거나, 문서 전체 상태가 개별 블록 상태와 독립적이라면 그 의미를 헤더에 명시한다.
