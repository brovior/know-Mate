## 총평
두 변경 모두 관측된 문제의 원인과 해결 방향을 합리적으로 연결하며, 특히 명시적 Qt 종료와 LanceDB 컬럼 projection은 적절한 선택이다. 다만 purge 상태 머신에는 정상적인 `next_retry_ts`를 비정상 미래 시각으로 처리하는 모순과, 백오프 만료 후에도 성공 스킵에 가로막혀 재시도되지 않는 결함이 있다. 또한 종료 모델은 QThread 외 보조 스레드까지 종료된다는 전제가 문서화되지 않아 “프로세스가 반드시 끝난다”는 요구를 완전히 입증하지 못한다. purge 메타 저장 실패 시 장애 증폭을 막는 정책도 보완해야 하므로, 현재 상태로는 구현 착수 전에 설계 수정이 필요하다.

**판정**: REQUEST_CHANGES

## 동의하는 결정
- `quitOnLastWindowClosed(False)`와 `_shutdown()` 말단의 명시적 `QApplication.quit()`을 결합한 것은 트레이 앱의 종료 의미를 창 가시성과 분리하는 적절한 결정이다.
- `_shutdown()` 앞 단계의 실패와 무관하게 최종 종료를 시도하면서, 실행 중 워커에는 기존 강제 종료 에스컬레이션을 유지한 방향은 FR-1과 FR-4의 우선순위를 잘 반영한다.
- purge 조회를 `file_path` projection 및 Arrow 직접 순회로 제한하고, 실제 컬럼 pushdown 검증을 채택 전제조건으로 둔 것은 메모리 문제의 근본 원인을 직접 해결한다.
- purge 스킵에 24시간 강제 reconciliation 상한을 두고, 메타 부재·손상을 보수적으로 “스킵 불가”로 처리한 것은 성능과 복구 가능성 사이의 균형이 타당하다.
- 기존 `index_state.json` 스키마를 변경하지 않고 원자 교체되는 sidecar를 사용한 결정은 마이그레이션 및 기존 소비자 회귀 위험을 줄인다.

## 지적 사항

### [B-1] 정상적인 백오프 시각을 비정상 미래 시각으로 판정함 (Blocker)
- **관점**: 7. 동시성, 8. 장애 대응
- **지적**: `next_retry_ts`는 정의상 `now + 백오프`인 미래 시각인데, 설계는 미래 시각 필드를 비정상으로 보고 해당 억제를 무시하도록 규정한다. 이대로 구현하면 실패 백오프가 항상 무력화될 수 있다.
- **근거**: A-0002 데이터 흐름은 “`next_retry_ts`는 미래값이면 비정상 → 해당 억제 무시”라고 한 직후 `now < next_retry_ts`이면 DB 조회 없이 반환하도록 정의한다. ADR-0002 역시 “미래 시각 필드는 해당 억제/스킵을 무시”한다고 포괄적으로 서술한다. 정상 실패 직후 저장되는 `next_retry_ts=now+30분`은 이 두 조건을 동시에 만족하므로 판정 규칙이 모순되며, R-0002 NFR-3의 장애 증폭 방지를 보장할 수 없다.
- **대안**: 시각 필드별 검증 규칙을 분리한다. `last_purge_ts`는 허용 오차를 넘는 미래값이면 무효로 처리하되, `next_retry_ts`는 `now < value <= now + configured_backoff + clock_skew_tolerance` 범위의 미래값을 정상으로 인정한다. 지나치게 먼 미래값만 손상으로 간주하고, 시스템 시각 역행 대응에는 저장 시각이나 백오프 길이도 함께 저장해 상한을 적용한다.

### [B-2] 백오프 만료 후 실패 purge가 성공 스킵에 가로막힘 (Blocker)
- **관점**: 7. 동시성, 8. 장애 대응
- **지적**: `failed_sig`의 백오프가 만료된 후 실패 재시도를 강제하는 분기가 없다. 기존 성공 메타가 남아 있으면 정상 성공 스킵 조건이 성립하여, 기본 30분 후 재시도한다는 상태 머신의 의도와 달리 최대 24시간 동안 재시도되지 않을 수 있다.
- **근거**: 예를 들어 op_sig `S`로 최근 성공한 뒤, 처리 건이 있는 사이클에서 purge가 실패하면 `failed_sig=S`, `next_retry_ts=now+30분`이 기록되지만 `reconciled_sig=S`와 최근 `last_purge_ts`는 그대로 남는다. 30분 후 처리 0건인 사이클에서는 실패 백오프 조건이 더 이상 반환하지 않지만, 바로 다음 성공 스킵 조건인 `reconciled_sig==S && 처리 0건 && 24h 미경과`가 참이 되어 purge가 실행되지 않는다. 이는 ADR의 “일시적 예외 기본 30분 백오프” 및 R-0002 FR-3의 백오프 간격 재시도 의미와 어긋난다.
- **대안**: 판정 순서를 `blocked → 유효한 실패 백오프 → 백오프가 만료된 동일 failed_sig면 강제 실행 → 성공 스킵 → 일반 실행`으로 명시한다. 또는 실패 시 해당 op_sig의 성공 스킵 자격을 별도 플래그로 무효화하고, 다음 성공 완료 때만 복구한다. 이 전이를 단위 테스트에 포함해야 한다.

### [M-1] QThread 외 보조 스레드 종료 계약이 없어 프로세스 종료 보장이 불완전함 (Major)
- **관점**: 2. 숨은 가정, 7. 동시성, 8. 장애 대응
- **지적**: A-0001은 `worker.isRunning()`만으로 hard exit 여부를 결정하지만, 제공된 구조상 CollectorWorker 내부에는 생산자 스레드와 워치독 타이머가 존재한다. QThread가 종료되더라도 non-daemon 보조 스레드가 남는다면 `app.exec()` 반환 후에도 Python 프로세스가 잔존할 수 있다.
- **근거**: 참조 구조에는 “Scanner → 생산자 스레드 → 큐 → 소비자”가 명시되어 있다. 특히 `QThread.terminate()`가 성공해 `worker.isRunning()==False`가 된 경우, 워커가 내부 생산자 스레드를 정상 join하는 finally를 실행하지 못할 가능성이 있다. A-0001의 종료 판정은 이러한 보조 실행 단위를 점검하지 않으므로 R-0001 FR-1/FR-4와 AC-3의 프로세스 소멸을 문서만으로 입증하지 못한다.
- **대안**: CollectorWorker의 종료 계약에 “생성한 모든 non-daemon 스레드를 추적하고 정상 경로에서 join한다”를 추가한다. 강제 terminate 경로에서는 보조 스레드 잔존 여부를 확인할 수 있는 `has_live_aux_workers()` 같은 계약을 두고, 하나라도 남으면 최종 기한 내 `hard_exit`로 전환한다. 워치독 타이머는 daemon임을 테스트하거나 명시적으로 취소·join해야 한다.

### [M-2] 실패·차단 메타 저장 실패 시 매 사이클 전건 조회가 재발할 수 있음 (Major)
- **관점**: 8. 장애 대응, 6. 보안, 9. 운영 복잡도
- **지적**: 설계는 purge 성공 후 메타 저장 실패의 멱등 재실행만 다루며, purge 예외 또는 대량삭제 차단 후 `failed_sig`·`blocked_sig` 저장이 실패하는 경우를 정의하지 않는다. 이 경우 다음 60초 사이클마다 projection 전건 조회와 동일 실패·차단이 반복될 수 있다.
- **근거**: R-0002 NFR-3은 purge 실패와 대량삭제 차단이 지속되어도 매 사이클 전건 조회가 반복되지 않아야 한다. 그러나 sidecar가 권한, 디스크 공간, 백신 잠금 등의 이유로 계속 저장되지 않으면 프로세스 재시작 전은 물론 동일 프로세스 안에서도 억제 상태가 보존되지 않는 설계가 될 수 있다. 대량삭제 차단 알림도 매번 반복될 여지가 있다.
- **대안**: sidecar 저장과 별개로 프로세스 내 메모리에 실패·차단 상태를 즉시 반영하여 현재 실행 중에는 억제를 유지한다. 저장 실패는 ERROR로 관측 가능하게 하고 제한된 주기로 메타 저장만 재시도한다. `failed_sig`/`blocked_sig` 저장 실패 테스트를 추가해 다음 사이클의 DB 조회 및 UI 알림이 반복되지 않는지 검증한다.

### [m-1] 경로 정규화만으로 안전한 소속 판정 방식이 충분히 정의되지 않음 (Minor)
- **관점**: 2. 숨은 가정, 6. 보안
- **지적**: 서명과 소속 판정이 같은 정규화 결과를 사용한다는 원칙은 좋지만, 정규화된 문자열 사이의 디렉터리 소속 판정 알고리즘이 명시되지 않았다.
- **근거**: 단순 접두사 비교를 사용하면 `C:/watch`가 `C:/watch-old/file`을 포함한다고 오판할 수 있다. purge는 실제 삭제로 이어질 수 있으므로 경로 경계, 드라이브가 다른 경우, 루트 경로, UNC 경로를 명시적으로 처리해야 한다. junction/symlink를 문자열상 별도 실체로 취급할지도 현재 문서에는 UNC/매핑 드라이브 외에는 정의되어 있지 않다.
- **대안**: `os.path.commonpath([file_path, root]) == root`와 드라이브 불일치 예외 처리 등 경계 인식 판정을 계약으로 명시한다. `C:\foo`/`C:\foobar`, 루트 폴더, UNC, 중첩 watch folder, junction 경로에 대한 삭제 대상 회귀 테스트를 추가한다.

### [m-2] 종료 단계 예외 테스트가 정상 순서 테스트보다 좁게 정의됨 (Minor)
- **관점**: 8. 장애 대응
- **지적**: A-0001은 각 단계 독립 예외와 최종 quit 보장을 핵심 정책으로 삼지만, 검증 계획은 주로 정상 호출 순서와 실기 경로에 집중되어 있다.
- **근거**: 실제 결함 위험은 `scheduler.stop()`, tray hide, `stop_worker()`, `worker.isRunning()` 또는 `quit()` 접근 중 하나가 예외를 발생시킬 때 후속 단계가 실행되는지에 있다. 특히 문서가 “예외와 무관하게 quit”을 요구하므로 이 부분은 명시적인 수용 테스트가 필요하다.
- **대안**: 각 단계가 개별적으로 예외를 던지는 매개변수화 테스트를 추가하고, 워커 미실행이면 quit이 호출되는지, 워커 실행 중이면 hard exit가 호출되는지, quit과 hard exit가 동시에 호출되지 않는지를 검증한다.

## 확인 필요
- 배포 고정 LanceDB 버전에서 `table.to_lance().to_table(columns=["file_path"])` 계열 API가 공개·안정 API인지, 실제 스캔 단계에서도 vector/text 컬럼을 읽지 않는지 확인 필요.
- CollectorWorker가 생성하는 생산자 스레드의 daemon 여부, 정상 취소 시 join 여부, `QThread.terminate()` 시 잔존 가능성을 소스 코드로 확인해야 한다.
- `max_delete_ratio`, 강제 주기, 백오프 설정이 NaN·Infinity·음수·과도한 값으로 들어오지 않도록 config 로드 단계에서 범위 검증되는지 확인 필요.
