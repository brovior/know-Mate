## 총평
두 ADR 모두 문제의 원인과 대안을 비교적 명확히 설명하며, 명시적 `quit()` 도입과 LanceDB 컬럼 projection 자체는 타당한 방향이다. 특히 종료 정리 순서를 보존한 점과 purge 성공 시에만 reconciliation 메타를 갱신하는 결정에 동의한다. 다만 A-0002의 24시간 강제 purge는 유휴 사이클의 메모리를 인덱스 크기와 무관한 상수 수준으로 제한한다는 승인 요구사항과 직접 충돌하며, `file_path` 전건 projection도 해당 사이클에서는 O(N) 메모리를 사용한다. 또한 purge 차단·실패가 지속되면 매 60초마다 전건 projection과 재시도가 반복될 수 있어 이번 변경의 운영 목적을 훼손한다.

**판정**: REQUEST_CHANGES

## 동의하는 결정
- 트레이 앱에서 `quitOnLastWindowClosed(False)`와 명시적 `QApplication.quit()`을 함께 사용하는 결정에 동의한다. 창 가시성에 따른 암묵 종료 규칙을 제거하고 종료 의도를 한 지점에서 명확히 표현한다.
- `quit()`을 `_shutdown()`의 마지막 단계에 배치하고 앞 단계 예외와 무관하게 호출되도록 보장하는 방향은 R-0001의 데이터 무결성 요구와 부합한다.
- purge 조회에서 벡터와 암호문을 제외하고 `file_path`만 projection하는 것은 불필요한 대용량 할당을 직접 제거하는 적절한 최적화다.
- purge가 성공적으로 완료된 경우에만 `reconciled_sig`와 `last_purge_ts`를 갱신하고, 메타 손상·부재 시 스킵하지 않는 보수적 정책에 동의한다.
- 메타를 기존 `index_state.json`과 분리하고 tmp→replace로 교체하는 결정은 기존 스키마 소비자에 대한 회귀 위험을 줄인다.

## 지적 사항

### [B-1] 24시간 강제 purge와 전건 projection이 승인된 메모리 요구를 위반함 (Blocker)
- **관점**: 1. 요구사항 누락, 4. 확장성, 5. 성능
- **지적**: A-0002는 24시간마다 유휴 상태에서도 purge를 강제 실행하며, 이때 모든 청크의 `file_path`를 메모리에 적재한다. 벡터를 제외해 상수 계수는 크게 줄지만, 메모리 사용량은 여전히 청크 수와 경로 길이에 비례하는 O(N)이다.
- **근거**: R-0002 NFR-1은 “유휴 방치 시(변경 0건) 사이클당 추가 메모리 할당이 인덱스 크기와 무관하게 상수 수준”일 것을 요구한다. 그러나 A-0002는 “24h 경과 시 0건이어도 purge를 1회 실행”하고 `file_path` 전건 projection을 수행한다. 따라서 24시간마다 발생하는 유휴 사이클은 명시된 NFR을 만족하지 않는다. 향후 청크가 10배 증가하면 벡터 메모리는 피하더라도 경로 Arrow 테이블과 pandas 변환 비용은 함께 증가한다.
- **대안**:
  1. 요구사항을 “스킵되는 일반 유휴 사이클은 O(1), 주기적 reconciliation은 O(N)이되 벡터·원문 크기와 무관하고 측정 상한을 충족”하도록 수정하고, 최대 청크 수별 메모리 수용 기준을 명시한다.
  2. NFR-1을 그대로 유지해야 한다면 `file_path`를 고정 크기 배치로 스트리밍하거나, LanceDB 측 조건 삭제/고유 경로 조회 등 DB pushdown을 사용해 애플리케이션 최대 메모리를 배치 크기로 제한한다.
  3. 최소한 pandas 변환이 필요 없다면 Arrow 컬럼을 직접 순회하여 Arrow→pandas 중복 표현과 추가 할당을 제거한다.

### [M-1] purge 지연 정책이 FR-3 및 AC-2와 일관되지 않음 (Major)
- **관점**: 1. 요구사항 누락
- **지적**: 승인된 요구사항은 purge 기능과 state-DB 불일치 복구가 “동작 변화 없이” 유지된다고 규정하지만, 설계는 불일치 복구를 최대 24시간 지연시킨다. 또한 AC-2는 “변경 0건 + watch_folders 불변 사이클”이면 DB 조회가 발생하지 않는다고 절대적으로 기술되어 있으나, 24시간 강제 reconciliation 사이클에서는 조회가 발생한다.
- **근거**: R-0002 FR-3 및 AC-2와 ADR-0002/A-0002의 강제 reconciliation 정책이 문언상 동시에 성립할 수 없다. ADR 결과에는 복구 지연을 감수하는 트레이드오프로 명시했으므로, 이는 구현 세부가 아니라 승인 요구의 변경이다.
- **대안**:
  1. R-0002 FR-3에 허용 가능한 복구 지연 상한을 24시간으로 명시하고, AC-2를 “마지막 성공 purge 후 24시간 미만” 조건까지 포함하도록 수정한 뒤 다시 승인한다.
  2. 요구사항 변경이 허용되지 않으면 매 사이클 reconciliation을 유지하되 배치 처리나 DB-side predicate로 메모리 비용을 제한한다.
  3. 외부 DB 변경이 지원되지 않는 불변식이라면 이를 요구사항에 명시하고, 주기적 reconciliation 자체를 제거하거나 별도의 진단/복구 작업으로 분리한다.

### [M-2] 차단·실패 시 매분 전건 재시도하는 핫 루프 가능성 (Major)
- **관점**: 5. 성능, 8. 장애 대응, 9. 운영 복잡도
- **지적**: purge 예외 또는 대량삭제 차단 시 메타를 전혀 갱신하지 않아 다음 60초 사이클에서 즉시 다시 전건 projection한다. 원인이 지속되면 큰 인덱스를 계속 읽고 동일 경고를 반복하는 장애 증폭 루프가 된다.
- **근거**: A-0002는 실패·차단 시 “다음 사이클 자동 재시도”한다고 규정한다. 대량삭제 차단은 설정이나 watch folder 구성이 바뀌기 전까지 계속될 가능성이 높고, 일시적이지 않은 DB 오류도 존재할 수 있다. 이 경우 유휴 메모리와 I/O 문제를 줄이려던 변경이 오히려 매분 반복되고 로그/UI 알림도 폭주할 수 있다.
- **대안**:
  1. 성공 메타와 별도로 `last_attempt_ts`, `failed_sig`, 실패 사유를 저장하고 지수 백오프 또는 설정 가능한 재시도 간격을 적용한다.
  2. 대량삭제 차단은 동일 `op_sig`에 대해 자동 재시도하지 않고, 구성·차단율 변경이나 사용자 승인 시에만 재실행한다.
  3. 일시적 예외만 제한 횟수로 재시도하고, 반복 실패 후에는 장기 백오프하면서 UI에 명시적인 “정리 필요” 상태를 노출한다. `reconciled_sig`는 성공 전까지 갱신하지 않아야 한다.

### [M-3] projection API 미확정 상태에서 FR-1 준수 폴백이 보장되지 않음 (Major)
- **관점**: 2. 숨은 가정, 5. 성능
- **지적**: 핵심 구현 API가 아직 검증되지 않았고, 실패 모드의 폴백 설명도 전체 테이블 로드로 되돌아갈 가능성을 배제하지 않는다.
- **근거**: A-0002는 `table.to_lance().to_table(columns=["file_path"])` 계열을 “1순위로 검토”하고 설치 버전 확인 후 확정한다고 한다. 동시에 실패 모드에는 기존 `to_arrow().to_pandas()`로 폴백한다고 표현되어 있는데, projection이 적용되지 않은 기존 호출이라면 R-0002 FR-1과 NFR-1을 다시 위반한다. `to_lance()`가 안정적인 공개 API인지와 실제 column pushdown 여부도 문서만으로는 확인되지 않는다.
- **대안**:
  1. ADR 채택 전에 배포에 고정된 LanceDB 버전에서 지원되는 공개 projection API와 반환 형식을 확정하고 문서에 기록한다.
  2. projection 미지원 시 조용히 전체 로드하지 말고 구현 또는 시작 단계에서 명확히 실패시키거나 해당 최적화를 비활성화하면서 경고한다.
  3. 테스트에 결과 컬럼 검증뿐 아니라 실제 스캔된 컬럼 또는 메모리 사용량 검증을 포함해, 결과에서 컬럼을 버리기만 하고 저장소에서는 전 컬럼을 읽는 가짜 projection을 방지한다.

### [m-1] 벽시계 역행 시 24시간 복구 상한이 깨질 수 있음 (Minor)
- **관점**: 2. 숨은 가정, 8. 장애 대응
- **지적**: `now - last_purge_ts < 24h` 판정은 시스템 시간이 뒤로 조정될 경우 예상보다 오래 스킵할 수 있다.
- **근거**: 메타는 프로세스 재시작을 넘어 유지되어 단조시계를 사용할 수 없고, Windows 시간 동기화나 수동 시각 변경이 가능하다. 음수 경과시간을 그대로 “24시간 미만”으로 처리하면 문서가 주장하는 최대 24시간 복구 상한이 성립하지 않는다.
- **대안**: UTC epoch를 저장하되 `now < last_purge_ts`이면 메타를 비정상으로 간주하여 즉시 purge하고, 비정상적으로 먼 미래 timestamp에 대한 테스트를 추가한다.

### [m-2] op_sig 직렬화 형식이 충분히 규정되지 않음 (Minor)
- **관점**: 2. 숨은 가정, 7. 동시성
- **지적**: 정렬·정규화와 SHA-256은 명시됐지만, 해시 입력의 필드 경계와 직렬화 형식이 정의되지 않았다.
- **근거**: 단순 문자열 결합은 경로와 설정값 조합에 따라 모호한 입력을 만들 수 있으며, 구현 변경 시 동일 설정의 서명이 불필요하게 달라질 수 있다. “UTF-8 직렬화”만으로는 프로세스 간 안정성을 완전히 규정하지 못한다.
- **대안**: 스키마 버전을 포함한 canonical JSON을 사용하고(`sort_keys=True`, 고정 separator, UTF-8), 경로 배열·`dry_run`·`max_delete_ratio`를 타입이 보존되는 별도 필드로 직렬화한다.

### [m-3] 종료 완료 수용 기준에 시간 상한이 없음 (Minor)
- **관점**: 1. 요구사항 누락, 8. 장애 대응
- **지적**: R-0001은 프로세스가 종료되어야 한다고 규정하지만, 언제까지 종료되어야 성공인지 정의하지 않는다.
- **근거**: 정상 워커 대기 8초와 terminate 대기 3초가 존재하므로 실기 검증에서 즉시 확인할지, 최대 11초 이상 기다릴지에 따라 결과 판정이 달라질 수 있다. “반드시 종료”만으로는 회귀 테스트와 릴리스 체크리스트가 결정적이지 않다.
- **대안**: 유휴 종료와 일반 종료는 예를 들어 3초 이내, 행오버 에스컬레이션 경로는 설정된 wait 합계와 여유 시간을 포함한 명시적 상한 이내로 AC를 구분한다.

## 확인 필요
- 소스 코드가 제공되지 않아 현재 트레이 [종료]와 `close_action=quit` 경로가 실제로 모두 `closeEvent → _shutdown()`으로 수렴하는지, `_shutdown()`에 재진입 방지 가드가 있는지 확인이 필요하다.
- 설치·배포된 정확한 LanceDB 버전과 해당 버전에서 공개적으로 지원되는 전건 column projection API, 그리고 실제 저장소 레벨 column pushdown 여부를 확인해야 한다.
- `_purge_removed_folders`가 pandas DataFrame을 반드시 요구하는지, Arrow 배열 직접 처리나 고정 크기 배치 처리가 가능한지 확인이 필요하다.
- 외부 도구 또는 향후 공용 DB 캐시 로직이 로컬 chunks DB를 변경할 수 있는지 확인해야 한다. 불가능한 불변식이라면 24시간 강제 reconciliation의 필요성과 복잡도를 재평가할 수 있다.
