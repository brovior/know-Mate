## 총평
두 ADR 모두 문제 원인을 구체적으로 설명하고, 종료 경로 수렴과 purge projection·상태 전이를 명확히 설계한 점은 타당하다. 특히 A-0002의 억제 판정 순서와 실패 시 성공 서명 해제는 재시도 무력화를 방지하는 좋은 결정이다. 다만 사용자 편집이 가능한 `max_delete_ratio`의 유효성 검증과 fail-closed 정책이 설계에 없어, 비정상 값이 대량삭제 차단기를 우회할 수 있다. 또한 강제 종료 후 LanceDB 손상 가능성을 인정하면서도 다음 실행에서 이를 식별하거나 사용자에게 복구를 안내할 운영 경로가 부족하다.

## 동의하는 결정
- `quitOnLastWindowClosed(False)`와 `_shutdown()`의 명시적 종료 수렴점을 함께 적용한 것은 트레이 앱의 창 가시성과 프로세스 수명을 분리하므로 적절하다.
- 워커 실행 여부를 확인한 뒤 `quit()` 또는 `hard_exit` 중 하나를 선택하고, 앞 단계 예외와 무관하게 최종 판정에 도달하도록 한 구조에 동의한다.
- purge에서 벡터·암호문을 제외하고 `file_path`만 projection하는 것은 관측된 메모리 문제의 직접 원인을 제거하는 결정이다.
- 차단 → 백오프 → 성공 스킵 → 실행 순서와, 실패 시 `reconciled_sig`를 해제하는 상태 전이는 재실행 안전성을 세심하게 다룬 설계다.
- sidecar 저장 실패와 메모리 상태 반영을 분리해 현재 프로세스의 반복 조회·반복 알림을 막고, 재시작 시 보수적으로 재실행하는 정책도 purge의 멱등성을 전제로 합리적이다.

## 지적 사항

### [B-1] 대량삭제 차단율의 유효성 검증과 fail-closed 정책 누락 (Blocker)
- **관점**: 2. 숨은 가정, 6. 보안
- **지적**: `max_delete_ratio`가 유한한 `0~1` 범위의 실수라는 전제만 있고, 비정상 설정값을 어디서 어떻게 차단할지 정의되어 있지 않다. 삭제 안전장치에 사용되는 값이므로 단순 기본값 폴백보다 purge 차단이 필요한 fail-closed 입력이다.
- **근거**: 설정 파일은 사용자가 직접 편집할 수 있는데, A-0002는 이 값을 그대로 op_sig와 대량삭제 판정에 사용한다. YAML의 `.nan` 등이 `NaN`으로 전달되면 Python에서 `delete_ratio > max_delete_ratio` 비교가 거짓이 되어 대량삭제 차단기가 우회될 수 있고, `>1` 값도 의도한 삭제 상한을 사실상 무력화한다. 또한 기본 `json.dumps()`는 `NaN`을 허용하므로 문서가 주장하는 canonical JSON도 엄밀히 성립하지 않는다. `dry_run=false`인 환경에서는 실제 대량삭제로 이어질 수 있어 안전 불변식 위반이다.
- **대안**: purge 진입 전에 `math.isfinite(value) and 0 <= value <= 1`을 검증하고, 실패하면 DB 조회·삭제 없이 차단 상태와 ERROR/UI 알림을 남긴다. op_sig 직렬화에는 `allow_nan=False`를 사용하고, `purge_force_reconcile_sec`와 `purge_backoff_sec`도 유한한 양수 범위로 검증한다. 이미 공통 config 검증기가 존재한다면 해당 보장을 A-0002에 명시하고 직접 편집·`NaN`·무한대·범위 초과 회귀 테스트를 추가해야 한다.

### [M-1] 강제 종료 후 인덱스 손상의 탐지·운영 복구 진입점 부족 (Major)
- **관점**: 8. 장애 대응
- **지적**: 최악의 경우 인덱스 폴더 삭제 후 전체 재인덱싱이 가능하다는 복구 방법은 제시했지만, 실제 사용자나 운영자가 언제 그 복구를 수행해야 하는지 판별하는 경로가 없다.
- **근거**: A-0001은 LanceDB `add/delete/optimize` 중 `terminate()` 또는 `os._exit()`가 발생했을 때 영향 범위와 커밋 원자성이 미확정이라고 명시한다. 다음 실행에서 DB open이 명시적으로 실패하면 발견할 수 있지만, 부분 반영이나 논리적 불일치로 열리면서 검색 결과만 누락되는 경우에는 현재 설계의 로그·검증 계획으로 식별하기 어렵다. “항상 유효한 수동 복구 방법”만으로는 장애 감지와 복구 안내가 완결되지 않는다.
- **대안**: 추측성 자동 복구 대신 최소한의 dirty-shutdown 표식을 도입한다. 워커 강제 종료 에스컬레이션 직전에 원자적으로 표식을 기록하고, 정상 종료 시 제거하며, 다음 시작 시 표식이 있으면 DB 기본 무결성/open 검사를 수행한 뒤 사용자에게 재인덱싱 선택지를 안내한다. 표식 도입도 보류한다면 배포 전에 LanceDB 강제 종료 실험으로 실제 실패 양상을 확인하고, 탐지 가능한 예외·로그 패턴과 인덱스 삭제/재구축 절차를 베타 운영 문서에 명시해야 한다.

### [m-2] sidecar 저장 실패 후 재실행 시점 설명 불일치 (Minor)
- **관점**: 8. 장애 대응
- **지적**: A-0002의 상태 전이 설명은 성공 메타 저장 실패 시 현재 프로세스에서는 메모리 캐시로 정상 스킵하고 재시작 후 재실행한다고 하지만, 실패 모드 표는 “다음 사이클 재실행”이라고 기술한다.
- **근거**: 같은 문서에서 저장 실패 후 DB 조회가 다음 사이클에 발생하는지, 재시작 뒤에만 발생하는지 상반되게 읽힌다. 이는 메모리 최적화의 핵심 동작과 테스트 기대값에 직접 영향을 준다.
- **대안**: 실패 모드의 감지/대응을 “현재 프로세스에서는 메모리 메타로 스킵 지속, 재시작 후 sidecar가 구버전이면 보수적으로 재실행”으로 통일하고 해당 동작을 테스트한다.

## 확인 필요
- 소스 코드가 제공되지 않아 `max_delete_ratio`, 강제주기, 백오프 값에 대한 기존 공통 설정 검증과 `json.dumps(..., allow_nan=False)` 적용 여부는 확인할 수 없다.
- R-0001·R-0002 요구사항 원문이 제공되지 않아 대량삭제 안전장치의 명시적 불변식과 강제 종료 후 복구에 대한 수용 기준은 확인이 필요하다.
- `hard_exit` 주입 함수가 반환하지 않는 `NoReturn` 계약인지 확인이 필요하다. 테스트 대역이 반환할 수 있다면 `stop_worker`에서 hard-exit 호출 후 최종 판정까지 이어져 “정확히 하나” 계약을 깨는 테스트상 모호성이 생길 수 있다.
