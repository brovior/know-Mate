## 총평
두 설계 모두 문제의 직접 원인을 식별하고 요구사항·ADR·검증 계획까지 연결한 점이 좋으며, 특히 종료 경로의 단일 수렴점과 purge의 projection 채택 전제조건은 타당하다. 다만 강제 종료 시 LanceDB 쓰기 안전성을 확인하지 않은 상태에서 영향 범위를 “그 사이클의 state 갱신뿐”으로 한정한 것은 문서 내부에서도 근거가 부족하며, 실제 손상 시 복구 계약도 아직 없다. purge 메타 상태 전이와 일부 ADR 문언도 구현자가 다르게 해석하지 않도록 정리할 필요가 있다. 핵심 방향을 재설계할 필요는 없지만 데이터 무결성 위험을 배포 전에 닫아야 한다.

**판정**: REQUEST_CHANGES

## 동의하는 결정
- 트레이 앱에서 `quitOnLastWindowClosed(False)`와 명시적 `quit()`을 조합하고 모든 사용자 종료 경로를 `_shutdown()`으로 수렴시킨 결정에 동의한다. 창 가시성에 의존하던 기존 결함을 직접 제거하며, 트레이 앱의 생명주기를 명시적으로 만든다.
- `stop_worker()` 예외 후에도 `isRunning()`을 다시 판정하고, 실행 중이거나 판정 불가이면 보수적으로 `hard_exit`를 선택한 것은 “종료가 반드시 완료되어야 한다”는 R-0001 FR-1 우선순위와 일치한다.
- purge에서 `file_path`만 projection하고 Arrow를 직접 순회하는 방향은 불필요한 벡터·암호문 적재를 제거하므로 성능 문제의 근본 원인에 대응한다. 지원되지 않는 버전에서 조용히 전체 로드로 폴백하지 않겠다는 조건도 요구사항 위반을 숨기지 않는 좋은 결정이다.
- 성공 스킵보다 실패 백오프와 대량삭제 차단을 먼저 판정하고, 실패 시 `reconciled_sig`를 해제하는 상태 전이는 재시도 누락과 장애 증폭을 함께 방지한다.
- sidecar 저장 실패 시 메모리 상태를 즉시 유지하면서 재시작 후 보수적으로 재실행하는 정책은 purge가 실제로 멱등하다는 전제 아래 운영 가용성과 안전성의 균형이 적절하다.

## 지적 사항

### [M-1] 강제 종료 시 LanceDB 무결성 영향과 복구 계약이 확정되지 않음 (Major)
- **관점**: 1. 요구사항 누락, 2. 숨은 가정, 7. 동시성, 8. 장애 대응
- **지적**: `QThread.terminate()` 또는 `os._exit()`가 LanceDB 쓰기 도중 실행될 때의 데이터 무결성 영향이 확인되지 않았는데도, A-0001은 정상적으로 오래 걸리는 추출까지 포함하여 “잃는 것은 그 사이클의 state 갱신뿐”이라고 영향 범위를 단정한다. 문서 뒤에서는 LanceDB 커밋 원자성이 버전·API 확인 대상이라고 인정하므로 두 설명이 서로 양립하지 않는다.
- **근거**: A-0001 정상 종료 계약은 “LanceDB 쓰기 도중이었다면 영향 범위는 LanceDB의 커밋 원자성에 의존한다”고 하면서 바로 이어 “잃는 것은 그 사이클의 state 갱신뿐”이라고 기술한다. `terminate()`는 임의 지점에서 QThread를 중단할 수 있고 `os._exit()`는 일반적인 정리 절차를 생략하므로, 고정 LanceDB 버전의 트랜잭션·파일 교체 보장이 확인되지 않으면 테이블 손상이나 미완료 쓰기를 배제할 수 없다. 현재 검증 계획에는 재시작 후 DB open·재인덱싱 테스트가 언급되지만, 실패 시 자동 복구·격리·전체 재구축 중 무엇을 수행할지 수용 기준이 없다.
- **대안**:
  1. “state 갱신만 유실”이라는 단정을 제거하고, 배포 고정 LanceDB 버전에서 write/add/delete/optimize 각 단계 강제 종료 후 재개 테스트를 수행해 보장 범위를 확정한다.
  2. 앱 시작 시 DB open 및 최소 읽기 검증을 수행하고, 실패하면 손상 인덱스를 격리한 뒤 로컬 원본에서 재구축하는 복구 경로를 설계한다. 자동 격리가 어렵다면 최소한 인덱싱을 중단하고 사용자에게 재구축 절차를 안내해야 한다.
  3. R-0001 AC에 “강제 종료 후 재시작 시 DB open 성공 및 재인덱싱으로 일관 상태 복구”를 추가하고, 확인 결과에 따라 graceful timeout 동안 LanceDB 쓰기 구간을 중단하지 않는 방식 등 종료 정책을 조정한다.

### [m-1] purge 메타의 결과별 전체 상태 전이가 불완전하게 명시됨 (Minor)
- **관점**: 2. 숨은 가정, 7. 동시성, 8. 장애 대응
- **지적**: 성공 결과는 실패·차단 필드를 모두 해제한다고 명시됐지만, 일시 실패와 대량삭제 차단 결과가 서로의 기존 필드 및 `last_purge_ts`를 어떻게 처리하는지는 완전하게 정의되지 않았다.
- **근거**: A-0002는 실패 시 `failed_sig`, `next_retry_ts`, `reconciled_sig` 해제만 규정하고 `blocked_sig` 처리 여부를 적지 않는다. 차단 시에는 `blocked_sig` 설정만 규정하고 기존 `failed_sig`, `next_retry_ts`, `reconciled_sig` 처리 여부가 없다. 판정 우선순위 덕분에 당장 오작동하지 않을 수 있지만, 설정을 A→B→A로 되돌리거나 상태가 실패→차단으로 전환될 때 오래된 억제 상태가 다시 유효해질 수 있어 구현별 해석 차이가 생긴다.
- **대안**: 결과별 메타 전이를 표로 고정한다. 예를 들어 성공은 모든 억제 상태 해제, 실패는 `failed_sig/next_retry_ts`만 설정하고 `blocked_sig/reconciled_sig` 해제, 차단은 `blocked_sig`만 설정하고 `failed_sig/next_retry_ts/reconciled_sig` 해제로 정의한다. 과거 동일 설정의 차단을 의도적으로 되살리는 정책이라면 그 의도를 요구사항과 테스트에 명시한다.

### [m-2] ADR-0002의 지연 설명이 24시간 강제 reconciliation과 불일치함 (Minor)
- **관점**: 1. 요구사항 누락, 9. 운영 복잡도
- **지적**: ADR-0002 결과에는 DB 고아 경로 복구가 “구성 변경 또는 파일 변경이 있는 사이클”까지 지연될 수 있다고 적혀 있으나, 결정과 architecture/requirements에는 최대 24시간 후 강제 reconciliation이 존재한다.
- **근거**: ADR의 해당 문언만 읽으면 유휴 상태에서 불일치가 무기한 유지될 수 있는 것으로 해석된다. 반면 R-0002 FR-3과 A-0002는 정상 상태에서 최대 24시간을 복구 시도 상한으로 규정한다.
- **대안**: ADR 결과를 “구성·파일 변경 시 즉시, 그 외에는 최대 강제주기 24시간까지 지연”으로 수정한다. 대량삭제 차단과 지속 장애는 이 상한의 명시적 예외라고 함께 적으면 요구사항과 완전히 정렬된다.

### [N-1] 문서 상태와 리뷰 이력 표기가 일관되지 않음 (Nit)
- **관점**: 9. 운영 복잡도
- **지적**: 두 ADR은 `Proposed`이면서 리뷰란에 “1~5차, 전건 종결”이라고 되어 있고, architecture는 이미 `Reviewed`이다. A-0001 본문에는 리뷰6·리뷰7 반영도 언급되지만 리뷰 이력은 5회로 기재되어 있다.
- **근거**: 이 상태에서는 ADR 결정이 아직 제안 단계인지, 독립 검증을 통과한 결정인지 추적하기 어렵다. 리뷰 파일 범위도 본문과 표가 다르다.
- **대안**: 실제 절차에 맞춰 ADR 상태와 리뷰 파일 목록을 갱신하고, architecture의 리뷰 차수·처리 요약과 동일한 기준으로 맞춘다.

## 확인 필요
- 배포 고정 LanceDB 버전에서 `table.to_lance().to_table(columns=["file_path"])` 계열 API가 공개·지원 API인지, 그리고 실제 스캔 단계에서 vector/text 컬럼을 읽지 않는지 확인 필요.
- LanceDB의 add/delete/optimize 수행 중 `QThread.terminate()` 또는 `os._exit()`가 발생했을 때 테이블 open 가능성, 커밋 원자성 및 공식 복구 절차를 확인해야 한다.
- 소스 코드가 제공되지 않았으므로 `_shutdown_done` 설정 시점, 각 종료 단계의 독립 `try` 처리, `quit()`/`hard_exit` 상호배타성 및 sidecar 메모리 선반영이 문서대로 구현 가능한지는 구현 리뷰에서 확인 필요.
