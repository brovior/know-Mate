## R-0001: 트레이 [종료]가 반드시 프로세스를 끝낸다  (상태: Approved)

### 배경 / 문제
베타 실사용에서 트레이 메뉴 [종료]를 눌러도 `AegisDesk.exe` 프로세스가 잔존한다(항상 재현).
트레이 아이콘은 사라져 겉보기엔 종료된 것처럼 보이지만 이벤트 루프·LanceDB·QWebEngine을 문
유령 프로세스가 남아 메모리를 계속 점유한다. 2026-07-23의 종료 확실화(`lifecycle.stop_worker`,
행오버 워커 강제 종료 + `os._exit`)는 **워커가 실행 중일 때만** 진입하는 안전망이라, 인덱싱이
돌지 않는 유휴 상태에서의 종료(트레이 종료의 대부분)는 커버하지 못했다.

### 기능 요구
- FR-1: 창이 트레이로 숨겨진 상태에서 [종료] → 프로세스가 종료된다.
- FR-2: 창이 보이는 상태에서 [종료] → 종료된다(기존 동작 회귀 없음).
- FR-3: `ui.close_action=quit`(X가 곧 종료) 설정 경로에서도 종료된다.
- FR-4: 워커가 COM 행오버 중일 때의 종료는 기존 에스컬레이션(정상→terminate→`os._exit`)을 유지한다.

### 비기능 요구
- NFR-1: 정상 종료 시 데이터 무결성 유지(state 저장·스케줄러 정지 후 종료).

### 제약
- Windows 전용 증상이나 수정은 크로스플랫폼 Qt API로만.
- CLAUDE.md 원칙8(수집기 QThread 단일)·기존 stop_worker 인터페이스 불변.

### 수용 기준 (Acceptance Criteria)
- [ ] AC-1: 창 숨김 상태에서 [종료] 후 작업관리자에 AegisDesk.exe가 없다.
- [ ] AC-2: 창 표시 상태 및 close_action=quit 경로도 동일.
- [ ] AC-3: 사외(pytest) 종료 경로 단위 테스트 통과.

### 명시적 비목표 (Out of scope)
- QWebEngineProcess 자식 프로세스의 정리 시퀀스 최적화(부모 종료 시 자연 종료에 의존).

## R-0002: 유휴 인덱싱 사이클의 반복적 전체 테이블 로드 제거  (상태: Approved)

### 배경 / 문제
유휴 자동 인덱싱이 기본 60초 간격으로 반복되는데, 매 사이클 `_purge_removed_folders`가
LanceDB chunks 테이블 **전체**를 `to_arrow().to_pandas()`로 로드한다 — 1024차원 float32 벡터
(청크당 4KB)와 암호화 원문까지 포함해서. 실제 필요한 것은 `file_path` 컬럼뿐이다. 변경 파일이
0건이어도 매분 수십 MB를 할당/해제하며, CPython/pyarrow가 힙을 OS에 즉시 반환하지 않아 RSS가
최고점에 눌러앉는다(베타에서 70MB 관측 — 인덱스가 수만 청크로 커지면 사이클당 수백 MB로 확대).

### 기능 요구
- FR-1: purge의 DB 조회는 필요한 컬럼(`file_path`)만 읽는다.
- FR-2: purge가 불필요한 사이클(판단 기준은 설계에서 정의)에는 전체 조회 자체를 생략한다.
- FR-3: purge의 본래 기능(제거된 watch_folder 청크 삭제, state-DB 불일치 복구, 대량 삭제 차단,
  dry_run)은 동작 변화 없이 유지된다.

### 비기능 요구
- NFR-1: 유휴 방치 시(변경 0건) 사이클당 추가 메모리 할당이 인덱스 크기와 무관하게 상수 수준.
- NFR-2: CLAUDE.md 원칙10(LanceDB API 규칙)과 충돌하지 않는 조회 방식.

### 제약
- LanceDB 접근은 메인 프로세스 QThread 워커에서만(원칙8).
- fake 모드 사외 테스트 통과(원칙4).

### 수용 기준 (Acceptance Criteria)
- [ ] AC-1: purge 조회 결과에 vector·text 컬럼이 포함되지 않음을 테스트로 검증.
- [ ] AC-2: 변경 0건 + watch_folders 불변 사이클에서 purge DB 조회가 발생하지 않음(스파이 검증).
- [ ] AC-3: watch_folder 제거 시나리오에서 기존과 동일하게 청크가 정리됨(회귀 테스트).

### 명시적 비목표 (Out of scope)
- 유휴 사이클 주기 자체의 변경(60초 유지 — 별도 논의).
- pyarrow/CPython의 OS 메모리 반환 강제(플랫폼 한계로 완화만 목표).
