# requirements.md — 요구사항 명세

| 상태 | 마지막 갱신 | 연결 문서 |
|---|---|---|
| Draft | (채택일로 갱신) | architecture.md, adr/ |

> **사용법**: 새 기능/변경 단위마다 아래 §N 블록을 추가한다. 상태는 Draft → (사람 승인) → Approved.
> Approved 전에는 architecture.md 설계를 시작하지 않는 것을 원칙으로 한다(요구가 흔들리면 설계도 흔들린다).
> 완료된 요구는 지우지 않고 "구현됨(커밋/PR 링크)"으로 남긴다 — 이 파일이 곧 요구 이력이다.

---

## 블록 템플릿

```markdown
## R-XXXX: <요구 제목>  (상태: Draft|Approved|구현됨)

### 배경 / 문제
(왜 이 요구가 생겼나 — 현상, 계기, 방치 시 비용)

### 기능 요구
- FR-1: (시스템이 무엇을 해야 하는가 — 검증 가능한 문장으로)
- FR-2: ...

### 비기능 요구
- NFR-1: (성능/보안/운영/호환 제약 — 수치나 판정 기준과 함께)

### 제약
- (불변식·환경 제약 — 예: CLAUDE.md §2 위반 불가, Windows/WSL 불가, 의존성 4개 고정)

### 수용 기준 (Acceptance Criteria)
- [ ] AC-1: (무엇이 관측되면 "됐다"고 판정하는가 — 테스트/실측으로 확인 가능한 형태)

### 명시적 비목표 (Out of scope)
- (이번에 하지 않는 것 — 리뷰어의 범위 오해를 막는다)
```
